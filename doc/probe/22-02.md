# Probe v4 — kimi-cli 1.37 resume semantics (methodology + raw notes)

**Probed**: 2026-04-23 ~05:45 local
**kimi-cli version**: 1.37.0 (`/Users/bing/.local/bin/kimi`)
**Platform**: darwin (macOS; `/tmp -> private/tmp` symlink in effect)
**Driver**: Claude Opus 4.7 (overnight autonomous execution; spec v1.1 triggered probe need)
**Author context**: kimi-plugin-cc v0.2 P2 spec v1.1 hypothesized (from kimi-cli PR #1716 source-read) that `kimi --print -r <bogus-uuid>` silently creates a ghost session. Probe v4 verifies (or falsifies) the hypothesis and extends probe v3 (kimi 1.36) with resume-semantics findings needed for spec v2 + plan.

## Methodology

```bash
cp -av ~/.kimi/kimi.json ~/.kimi/kimi.json.bak-probe-v4-20260423-054458
mkdir -p /tmp/kimi-probe-v4-<ts> /tmp/kimi-probe-v4-cross-<ts>
```

`ts` = `1776894306` (generation: `date +%s`).

Verified prior to probe:
- `/tmp/kimi-probe-v4-<ts>` was not in `~/.kimi/kimi.json.work_dirs`.
- `~/.kimi/sessions/<md5(cwd)>` was not pre-existing for either cwd.

### Step 1 — Seed session in cwd A (answers Q4.1, Q4.2, Q4.5)

```bash
cd /tmp/kimi-probe-v4-1776894306 && kimi --print -p "say just the word OK, nothing else" --quiet
```

Outcome:
- kimi stdout: `OK\n\nTo resume this session: kimi -r 236d5d09-0c3e-49af-b637-d03ae2ca194b`
- exit: 0
- `~/.kimi/kimi.json.work_dirs` appended: `{"path": "/private/tmp/kimi-probe-v4-1776894306", "kaos": "local", "last_session_id": "236d5d09-0c3e-49af-b637-d03ae2ca194b"}`
- `~/.kimi/sessions/4be2dad86b97966a68e6805446b784fa/236d5d09-.../` created with 3 files:
  - `context.jsonl` — 17,636 bytes, UTF-8 JSONL, first event `{"role":"_system_prompt","content":"You are Kimi Code CLI, ..."}`
  - `state.json` — 413 bytes, JSON object with keys `[version, approval, additional_dirs, custom_title, title_generated, title_generate_attempts, plan_mode, plan_session_id, plan_slug, wire_mtime, archived, archived_at, auto_archive_exempt, todos]`
  - `wire.jsonl` — 1,112 bytes

md5 cross-check: `md5(/private/tmp/kimi-probe-v4-1776894306) = 4be2dad86b97966a68e6805446b784fa` ✓ matches sessions dir; `md5(/tmp/kimi-probe-v4-1776894306)` = 32a755b56c94e19df700541c912af1e6 ✗ does NOT match. Confirms realpath-resolved storage.

UUID `236d5d09-0c3e-49af-b637-d03ae2ca194b`: `uuid.UUID(...).version == 4`, `variant == "specified in RFC 4122"`.

### Step 2 — Ghost experiment in cwd A (answers Q4.0, Q4.4)

Fabricate UUID: `uuidgen | tr '[:upper:]' '[:lower:]'` → `501c89f6-2566-4895-ae43-64b8c57ffe3e`. Verified not already on disk (`find ~/.kimi/sessions -name "501c89f6..."` returned no hits).

```bash
cd /tmp/kimi-probe-v4-1776894306 && kimi --print -r 501c89f6-2566-4895-ae43-64b8c57ffe3e -p "say just OK" --quiet
```

Outcome:
- kimi stdout: `OK\n\nTo resume this session: kimi -r 501c89f6-2566-4895-ae43-64b8c57ffe3e` — **note kimi echoes back the fabricated UUID as the resume target, not a fresh-generated one**
- exit: 0
- stderr: (empty)
- `~/.kimi/sessions/4be2dad86b97966a68e6805446b784fa/501c89f6-.../` created with full 3-file layout:
  - `context.jsonl` — 17,111 bytes
  - `state.json` — 390 bytes
  - `wire.jsonl` — 1,005 bytes
- `~/.kimi/kimi.json.work_dirs[path=/private/tmp/kimi-probe-v4-...]` `last_session_id` updated from `236d5d09-...` to `501c89f6-...` — kimi treats the ghost as the new "last" session

**Structural comparison (ghost vs real)**: identical file count, identical keys in state.json, context.jsonl starts with same system prompt, wire.jsonl has same shape. Zero user-visible indication that "resume" was actually "create-from-bogus-UUID". **Hypothesis CONFIRMED.**

### Step 3 — Cross-cwd experiment (answers Q4.3)

```bash
cd /tmp/kimi-probe-v4-cross-1776894306 && kimi --print -r 236d5d09-0c3e-49af-b637-d03ae2ca194b -p "say just OK" --quiet
```

(using the UUID seeded in cwd A, invoked from cwd B.)

Outcome:
- exit: 0
- Before: `~/.kimi/sessions/e98fe6a757630f40c0c9124c1e2e1157/` did not exist (cwd B fresh).
- After: `~/.kimi/sessions/e98fe6a757630f40c0c9124c1e2e1157/236d5d09-0c3e-49af-b637-d03ae2ca194b/` created with full 3-file layout. (Note: this is cwd B's md5, NOT cwd A's.)
- `~/.kimi/kimi.json.work_dirs` added: `{"path": "/private/tmp/kimi-probe-v4-cross-1776894306", "kaos": "local", "last_session_id": "236d5d09-..."}`
- The original cwd A session at `md5(A)/236d5d09-.../` was unaffected.

**Conclusion**: kimi does NOT check that the UUID "belongs to" the current cwd before creating the session dir. Cross-cwd resume triggers the same ghost behavior as Q4.0 with a bogus UUID. §7.3 spec decision (refuse cross-cwd resume wrapper-side) is strongly justified — without it, users would silently fork their sessions across directories.

### Step 4 — kimi-cli help scan (answers Q4.6)

`kimi --help | grep -iE '(session|resume|continue)'` produces:
- `--session, --resume -S, -r  TEXT  Resume a session. With ID: resume that session. Without ID: interactively pick a session.`
- `--continue -C  Continue the previous session for the working directory.`
- `export  Export session data.` (subcommand)

**Finding**: kimi 1.37 has a BUILT-IN `--continue / -C` flag that natively does what `/kimi:continue` is specced to do. This is a significant P2 design input — surface in 6-way review.

`kimi export --help` confirms it's a ZIP archive exporter (`kimi export [SESSION_ID]`), not a session lister or validator. No dedicated `sessions list` or `resume --check` subcommand exists.

## Derived findings → spec revisions needed (v1.2 ahead of 6-way review)

See `probe-results-v4.json` key `spec_revisions_needed_for_v1_2`. Summary:

1. `§4.4 resolveContinueTarget`: iterate `work_dirs` array (`.find(w => w.path === normalizedCwd)`), not object-index lookup. **This is a correctness bug in v1.1 pseudocode.**
2. `§4.4 validateResumeTarget`: detection predicate is `exists(dir) && exists(dir/context.jsonl) && size(dir/context.jsonl) > 0`.
3. `§2.4` assumptions: upgrade all three to "probe v4 confirmed" with precise shape notes.
4. `§9.2 BREAKING`: remove the hedging ("pending probe v4 Q4.0 — if probe confirms") — probe confirmed. Framing is "fixes ghost-session" unambiguously.
5. New `§12.2` open question: `/kimi:continue` design — wrapper explicit (current v1.1 design) vs. thin pass-through to `kimi -C` (probe-derived alternative). Surface in 6-way review for reviewer input.

## Known limitations / followups

- Did NOT probe the interactive-mode `kimi -r` (no TTY in probe environment). Source-read suggests interactive mode has the PR #1716 validation raise that print mode lacks, but this is spec-level documentation, not a gap that affects P2 correctness (P2 is one-shot print mode only).
- Did NOT probe `kimi -C` ghost behavior in a fresh cwd (whether it no-ops or creates a new ghost session). Surface as open question for 6-way review; if selected as P2 design, a probe v4.1 extension is needed.
- Did NOT probe concurrent-write race on `kimi.json` (another kimi running simultaneously). §7.6 "accept the race" still stands; probe for it only if 6-way review flags it as important.
- Did NOT probe what happens if wrapper passes `-w <explicit-realpath>` vs. just `cd`; both paths should produce the same md5 key, but not directly confirmed.

## Platform-specific notes

- macOS: `/tmp -> private/tmp` symlink is the reason kimi 1.37 canonicalized to `/private/tmp/...` in our probe. On Linux, `/tmp` is typically NOT a symlink, so paths would already be canonical. Spec §7.7 wrapper-side `resolveRealCwd()` is idempotent on paths that are already realpath-resolved, so the Linux behavior is covered by the same code path.
- macOS: `timeout` command not available in default `/usr/bin`. Probe used manual kill-on-error rather than timeout-bounded invocations. All Q4 kimi calls completed within ~10 seconds each.
