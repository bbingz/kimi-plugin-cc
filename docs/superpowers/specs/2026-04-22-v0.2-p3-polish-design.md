### New external contract: `KIMI_JOB_TTL_DAYS` env var

- Accepts positive integer (days). Same strict `/^\d+$/` parse as `KIMI_TIMEOUT_MS`.
- Invalid → stderr warning + fallback to default `7`.
- `0` is valid and means "never expire" (escape hatch).
- No other UI surface (no `/kimi:setup --job-ttl`); env-only keeps v0.2 scope tight.

### Invariant shift: state.json retention semantics

**Before**: `cleanupSessionJobs(cwd, sessionId)` on SessionEnd filtered out **all** jobs for the ended session (running + completed + cancelled). Completed-job state vanished the moment the Claude Code session closed.

**After**:
- SessionEnd keeps finished jobs (status ∈ `completed | failed | cancelled`); removes only `running | starting | queued` (which are orphaned when the session ends).
- `loadState` applies a TTL filter: jobs with `completedAt` older than `Date.now() - ttlMs` are purged on every load.
- Purge is **atomic via existing `updateState`** shape; no new concurrency primitive.

## 2. Item-level designs

### G1.C2 — error-shape unification (catch-wrapper path only)

**Scope clarified post-review**: the happy-path shapes of the 4 handlers are legitimately different (review returns `{verdict, findings[]}`; task returns `{jobId, status}`; ask streams JSONL) — we do NOT unify those. What we unify is the **outer catch-wrapper shape** that each handler uses when it hits an unexpected exception.

**Current** (verified via grep of `kimi-companion.mjs`):
- `runAsk` catch: `process.stdout.write(JSON.stringify({ ok: false, error: e.message }))` — line ~387
- `runReview` catch: same shape `{ok: false, error: e.message}` — line ~484
- `runAdversarialReview` catch: same `{ok: false, error: e.message}` — line ~484
- `runTask` catch: same `{ok: false, error: err}` — lines ~589, 600
- But each handler additionally synthesizes transport-error results (like `{error: "Unexpected error during review: <msg>"}` at line 442) with varying fields

The baseline `{ok: false, error: msg}` is already consistent; the drift is in the **transport-error records** that some handlers produce when they get a result object back from `callKimi*` with `result.ok === false`.

**Target**: one `errorResult(...)` helper in `job-control.mjs`, exported, signature:

```js
// Canonical transport-error envelope — use at every catch / transport-failure site
errorResult({ kind, error, status = null, stdout = "", detail = null })
// kind ∈ "ask" | "review" | "adversarial-review" | "task"
// (rescue is not in scope — it's an Agent-based subagent call, not a run* handler)
```

Review paths keep their richer `reviewError` (adds `verdict: "needs-attention"`, `findings: []`, `retry_used`, `truncation_notice`, `summary`) — **post-P3 `reviewError` internally composes `errorResult` + pipeline fields** so the base shape is present in every review error.

**Why now**: lets command `.md` renderers branch on one canonical field set (`ok`, `kind`, `error`) instead of probing per-command.

**Scope boundary**: does NOT change exit codes, stderr format, or user-facing error strings. Does NOT touch the happy-path return shapes. Pure catch-wrapper / transport-error consolidation.

### G1.C5 — `enrichJob` pure + apply split

**Current** (`job-control.mjs:359`):
```js
function enrichJob(job, workspaceRoot) {
  const enriched = { ...job };
  // (1) zombie detection — MUTATES state
  if (enriched.status === "running" && enriched.pid && !isProcessAlive(enriched.pid)) {
    enriched.status = "failed"; ...
    upsertJob(workspaceRoot, { ... });   // ← side effect during read-only status
  }
  // (2) elapsed time — pure
  // (3) log preview — reads file (IO, not state mutation)
  // (4) kind label — pure
  return enriched;
}
```

**Target split**:

```js
// Pure — takes pre-read content, returns enriched + "needs persist" signal
export function enrichJob(job, { logPreview = "", isAlive }) {
  const enriched = { ...job };
  let shouldPersistZombie = false;
  if (enriched.status === "running" && enriched.pid && !isAlive) {
    enriched.status = "failed"; enriched.phase = "failed";
    enriched.detail = "Process exited unexpectedly";
    shouldPersistZombie = true;
  }
  // elapsed + log preview + kind label — all pure now
  return { enriched, shouldPersistZombie };
}

// Thin IO wrapper — called by status / result paths
function enrichJobFromDisk(job, workspaceRoot) {
  const logPreview = readJobProgressPreview(...);
  const isAlive = isProcessAlive(job.pid);
  const { enriched, shouldPersistZombie } = enrichJob(job, { logPreview, isAlive });
  if (shouldPersistZombie) {
    upsertJob(workspaceRoot, { id: enriched.id, status: "failed", phase: "failed", pid: null });
  }
  return enriched;
}
```

**Caller migration**:
- `/kimi:status` call site uses `enrichJobFromDisk` (unchanged externally); internal zombie-persist path preserved.
- Future test calls `enrichJob` directly with mocked `{logPreview, isAlive}` inputs.

**Scope clarified post-review (Codex-M1)**: C5 does NOT make `/kimi:status` side-effect-free. `enrichJobFromDisk` still calls `upsertJob` for zombie persistence during status read. The claim is narrower — **C5 makes `enrichJob`'s body testable in isolation** (no `workspaceRoot` coupling, no hidden file reads), which P1's tests/ directory requires. Eliminating the zombie-persist-during-read side-effect is a separate v0.2+ refactor (requires splitting status into query+reconcile, breaks API shape) and is recorded in lessons.md §I.2 as deferred.

**Why now**: foundation for proper unit tests in P1 (tests/ directory lands there) + surfaces the "status-reads-mutate-state" anomaly for future explicit fix.

### G2.C8 — `maxDiffChars` parameterization in `runReviewPipeline`

**Naming correction post-review (Codex-M2)**: the existing constant is named `MAX_REVIEW_DIFF_BYTES` but the boundary is enforced in the companion using `context.content.length` (JS string length = UTF-16 code-unit count, i.e. chars, NOT bytes). Introducing a **new** parameter named `maxDiffBytes` would perpetuate the misnomer. We name the parameter truthfully: `maxDiffChars`. The existing exported constant `MAX_REVIEW_DIFF_BYTES` stays (internal consumers: `review.mjs:21`, `kimi.mjs:10, 19`) to avoid a rename ripple — we add a clarifying comment at its declaration.

**Current** (`review.mjs:210`, already-partial from 5-way review commit `54f2fd0` — notices were parameterized, bytes limit was not):
```js
export function runReviewPipeline({
  // ...
  truncationNotice = TRUNCATION_NOTICE,   // module constant
  retryNotice = RETRY_NOTICE,             // module constant
} = {}) { ... }
```

`MAX_REVIEW_DIFF_BYTES` is consumed by `callKimiReview` / `callKimiAdversarialReview` at their boundary, NOT passed through the pipeline. A sibling with a 64KB budget would pass a pre-truncated context but `truncationNotice` would still embed the "150 KB" string from kimi's default — and the **bytes/chars confusion** means even the number is slightly off for multi-byte content.

**Target**: add `maxDiffChars` as a pipeline parameter; default-derive `truncationNotice` from it.

```js
// review.mjs:13 — comment clarified (content unchanged)
export const MAX_REVIEW_DIFF_BYTES = 150_000; // NB: measured as JS string length
                                              // (UTF-16 code units, not UTF-8 bytes).
                                              // Name kept for back-compat; new callers
                                              // should think in "chars".

// review.mjs:210 — new parameter
export function runReviewPipeline({
  // ...
  maxDiffChars = MAX_REVIEW_DIFF_BYTES,                   // ← new; measured in chars
  truncationNotice = formatTruncationNotice(maxDiffChars), // ← derived from chars
  retryNotice = RETRY_NOTICE,
} = {}) { ... }
```

**Caller migration**: `callKimiReview` / `callKimiAdversarialReview` already pass no notice args; they now also pass no `maxDiffChars`, inheriting default. No signature change at the two wrappers. Sibling plugins with a different char budget pass `maxDiffChars` explicitly — and the default `truncationNotice` string re-derives from that value.

**Invariant**: `truncationNotice` and `maxDiffChars` stay in sync unless caller explicitly passes both. Spec-documented "pass both or neither for self-consistent output" (mirrored in review.mjs comment + risk table R5).

**Scope boundary**: the truncation site in `kimi-companion.mjs` (~line 417, 534) still uses `content.length` comparison. We do NOT switch to `Buffer.byteLength` in this PR — it changes the semantic (fewer chars fit before cap), which is a user-visible budget change that should be decided explicitly, not tacked into P3.

### G3.C3 — defensive large-prompt cap (rationale corrected)

**Rationale correction post-review (Codex-H3)**: the v1 spec framed this as "E2BIG / ARG_MAX prevention." That framing was technically wrong — kimi uses stdin for prompts (probe-results.json v3 confirms: `--input-format text` piped via child stdin), and **stdin has no `ARG_MAX` limit**. The kernel's `E2BIG` applies to argv+env, not pipe writes. So the original rationale doesn't hold.

**What this guard actually does**: provide a **predictable, explicit failure message** when a caller hands kimi a pathologically large prompt. kimi-CLI's own handling of multi-MB stdin is untested empirically (our Phase-0 probes went up to 150 KB). Until a probe establishes kimi's real upper bound, a defensive cap prevents:
- hanging writes on slow pipe consumers
- opaque OOM / truncation failures from kimi itself
- silent content dropping that produces misleading answers

**Target**: explicit pre-spawn length check in both `callKimi` and `callKimiStreaming`:

```js
const MAX_PROMPT_CHARS = 1_000_000; // ~1 MB assuming ASCII; much less for CJK.
                                    // Defensive cap only — not a kernel boundary.

// label: human-facing function name ("ask" | "review" | "adversarial-review" | "task")
// used only in the error message so users know which /kimi:* call to retry with less content
// kind: aligns with errorResult from C2 — same values as label here
function checkPromptSize(prompt, { kind, label }) {
  const chars = prompt.length; // char count matches how we already measure everywhere
  if (chars > MAX_PROMPT_CHARS) {
    return errorResult({
      kind,
      error: `prompt exceeds ${MAX_PROMPT_CHARS} chars (got ${chars}); trim context or split into multiple ${label} calls`,
      status: null,
      stdout: "",
      detail: `prompt-too-large: ${chars} chars > ${MAX_PROMPT_CHARS} char cap`,
    });
  }
  return null;
}
```

Called at the top of each `callKimi*`; returns early with the canonical `errorResult` shape (C2-aligned; Qwen-H1 fix). Two call sites:
- `callKimi` — invoked only by `/kimi:ask`; pass `{ kind: "ask", label: "ask" }`
- `callKimiStreaming` — invoked by `/kimi:task` foreground + background; caller passes `kind` via the existing `config` (default `"task"`); `label` derived from `kind`

**Scope boundary**: no streaming-time limit (cap is at boundary only). Not user-tunable — this is a defensive constant, not a budget knob. If a real user case hits it legitimately, P4 lessons.md §I.2 tracks as "revisit when probe establishes kimi's real stdin ceiling."

**Why 1 million chars**: 150 KB diff (our empirical max-tested) is ~150K chars in ASCII, comfortably below. 1M leaves 6-7x headroom for adversarial-review that combines big diff + verbose focus text + code excerpts. CJK content compresses worse (~3 bytes/char UTF-8) but we're measuring chars so the 1M boundary behaves consistently regardless of script.

### G4.C6 — SessionEnd TTL policy (+ `KIMI_JOB_TTL_DAYS`)

**Two changes**, separable but landed together:

#### (a) SessionEnd cleanup narrowed

**Current** (`session-lifecycle-hook.mjs` — function `cleanupSessionJobs` defined at line 66, filter statement at line 85):
```js
jobs: state.jobs.filter((j) => j.sessionId !== sessionId),  // removes ALL with this sessionId
```

**Target**: keep finished jobs; only remove orphaned in-flight jobs.

```js
jobs: state.jobs.filter((j) => {
  if (j.sessionId !== sessionId) return true;
  const finished = j.status === "completed" || j.status === "failed" || j.status === "cancelled";
  return finished; // keep finished even for ended session
});
```

Running jobs from an ended session get dropped (they're already orphaned — the session's `SIGTERM`-on-close leaves them effectively dead).

#### (b) TTL filter placement — split between `loadState` (read-view) and `updateState` (physical purge)

**Concurrency correction post-review (Codex-H1 + Gemini-M1 convergent)**: v1 spec proposed "TTL filter in `loadState` only," relying on incidental `updateState` calls to durably purge. That's unsafe: `session-lifecycle-hook.mjs:73-86` does unlocked `loadState(...) → saveState(...)` RMW (no `updateState` wrapping), so if TTL purge happens inside `loadState`, a hook running concurrently with a worker's `updateState` could overwrite state.json with a stale-minus-expired snapshot, losing the worker's updates.

**Corrected design** — separate concerns:

```js
// state.mjs — shared helpers

const DEFAULT_TTL_DAYS = 7;
function resolveTtlMs() {
  const raw = process.env.KIMI_JOB_TTL_DAYS;
  if (raw === undefined) return DEFAULT_TTL_DAYS * 24 * 60 * 60 * 1000;
  if (!/^\d+$/.test(raw)) {
    process.stderr.write(`[kimi] ignoring invalid KIMI_JOB_TTL_DAYS=${raw}; using default ${DEFAULT_TTL_DAYS} days\n`);
    return DEFAULT_TTL_DAYS * 24 * 60 * 60 * 1000;
  }
  const days = Number.parseInt(raw, 10);
  if (days === 0) return Infinity;            // explicit opt-out
  return days * 24 * 60 * 60 * 1000;
}

function filterExpired(jobs, ttlMs, nowMs) {
  if (ttlMs === Infinity) return jobs;
  return jobs.filter((j) => {
    if (!j.completedAt) return true;         // in-flight → keep
    const age = nowMs - new Date(j.completedAt).getTime();
    return age < ttlMs;
  });
}

// loadState — applies filter to the RETURNED OBJECT only; does NOT touch disk.
// Callers see a purged view; users see no expired jobs in /kimi:status output.
export function loadState(workspaceRoot) {
  const state = readStateFile(workspaceRoot);  // existing read logic
  return { ...state, jobs: filterExpired(state.jobs, resolveTtlMs(), Date.now()) };
}

// updateState — already wraps read+mutate+write in a lock. The physical
// purge happens here (not in loadState), so the write is atomic with
// any other mutation and can't race against unlocked RMW in hooks.
export function updateState(workspaceRoot, mutate) {
  // ... existing lock acquisition + atomic read
  const state = readStateFile(workspaceRoot);
  state.jobs = filterExpired(state.jobs, resolveTtlMs(), Date.now()); // ← new
  mutate(state);
  writeStateAtomic(workspaceRoot, state); // ← physically drops expired entries
  // ... release lock
}
```

**Properties**:
- Pure reads (`/kimi:status`, `/kimi:result`) see filtered view; disk unchanged.
- Any mutation (job create, complete, cancel, SessionEnd's own path if it uses `updateState`) atomically purges expired entries with the same write.
- The unlocked `loadState → saveState` path in `session-lifecycle-hook.mjs` is NOT affected — because SessionEnd only filters by sessionId, never touches the TTL logic on its write path. (A follow-up v0.2 task could migrate SessionEnd to `updateState`; out of P3 scope.)

**Stale-file concern**: `<jobId>.log` files and `<jobId>.json` files in the state dir don't get purged by the TTL. They're already swept by `cleanupOrphanedFiles` when they outlive their state record by 60s. Once TTL invalidates a job record (next `updateState` write), the next orphan-sweep disposes of the files.
