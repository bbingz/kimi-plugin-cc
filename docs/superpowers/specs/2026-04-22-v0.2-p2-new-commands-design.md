## 0. Context / Motivation

### 0.1 Why /kimi:continue + /kimi:resume

Today a Kimi user who wants to resume a prior conversation has two unattractive options:

1. `/kimi:ask --resume <uuid> <prompt>` — works, but (a) requires the user to already know the UUID, (b) **is hypothesized to have a silent ghost-session bug** (source-read from kimi-cli PR #1716; pending probe v4 verification — see §0.2): passing a bogus UUID may silently create an empty session at that UUID and treat it as a fresh start, without telling the user.
2. `/kimi:rescue --resume-last` — works, but `--resume-last` means "resume the most recent *rescue/task* candidate" (see `kimi-companion.mjs task-resume-candidate`), **not** "resume the most recent cwd session". Mismatched mental model.

There is no ergonomic way to say "keep working on what we were just doing in this repo." P2 fills that gap with two new slash commands:

- `/kimi:continue <prompt>` — resume the most recent session for the current working directory.
- `/kimi:resume <sessionId> <prompt>` — resume a specific session by UUID (scoped to the current cwd).

### 0.2 Hard constraint — pending probe v4 verification

**Hypothesis (source-read; not yet probe-verified)**: `kimi --print -r <bogus-uuid>` silently creates a ghost session at that UUID rather than rejecting. The interactive-mode `-r` validation raise introduced in kimi-cli PR #1716 does not fire in `--print` mode. Any P2 command that invokes `kimi -r` without wrapper-side pre-validation would inherit this bug.

**Evidence status**: This hypothesis originated from reading kimi-cli source (PR #1716 and `kimi_cli/cli/__init__.py`) — the existing on-disk `doc/probe/probe-results.json` (schema_version 3, 2026-04-20, kimi 1.36.0) does NOT include a resume/ghost experiment. Probe v4 (§2) will execute the `kimi --print -r <fab-uuid>` experiment from a throwaway cwd and either CONFIRM the hypothesis (spec remains) or FALSIFY it (spec motivation collapses; P2 must be re-scoped).

**Design posture under uncertainty**: Spec is written as if the hypothesis will be confirmed. The wrapper-side validation is also defensible as a general "input-sanity hardening" step even if kimi-cli turns out to reject bogus UUIDs, so the blast radius of the probe invalidating the hypothesis is limited to the prose of §0/§9 BREAKING notes — the code would still be correct, just less dramatic.

**If hypothesis confirmed**: every P2 resume entry point MUST validate the resume target before invoking kimi-cli. This shapes the error matrix (§6/§7).

### 0.3 Sibling-series significance

`/kimi:continue` and `/kimi:resume` are the **first** user-facing resume commands in the kimi-plugin-cc template series. Scan conducted 2026-04-22 at these literal paths:
- `/Users/bing/-Code-/gemini-plugin-cc/plugins/gemini/commands/ask.md` — no `--resume` or `--session` flag
- `/Users/bing/-Code-/minimax-plugin-cc/plugins/minimax/commands/ask.md` — no `--resume` or `--session` flag
- `/Users/bing/-Code-/qwen-plugin-cc/plugins/qwen/commands/ask.md` — explicitly declares `/qwen:ask` is "always a new conversation; resume belongs to /qwen:rescue --resume-last"

Only kimi's `/kimi:ask` uniquely has `--resume <sessionId>` today (which P2 removes — see §9). P2 sets the template that future sibling backports (gemini, minimax, qwen, eventual doubao) will mirror.

The sibling precedent is also why P2 removes the `/kimi:ask --resume` flag (see §9): three of three siblings have `ask` as "new-conversation-only"; keeping `/kimi:ask --resume` as a kimi-only special case would create permanent "why does kimi have this but nobody else?" documentation drag.

### 1.2 Out of scope (YAGNI)

- `/kimi:list-sessions` — deferred. Rationale: users who need a listing can `jq` `~/.kimi/kimi.json` or `ls ~/.kimi/sessions/<md5(cwd)>/`; no evidence yet that a first-class listing UX is required. Revisit after P2 ships.
- `--model <name>` flag on continue/resume — deferred. Resume inherits the session's original model (kimi-cli behavior); overriding mid-stream complicates validation. Revisit if users request it.
- `--stream` flag on continue/resume — deferred. Matches ask.md's rule: `--stream` is developer-only when invoking the companion directly; slash commands always go through the gated (non-stream) path.
- Cross-cwd resume — **explicitly disallowed** (see §7). Session-to-cwd is 1:1 via `md5(realCwd)` path; resuming a session that belongs to a different cwd would either fail silently in kimi-cli or create inconsistent state. Not worth the test matrix.
- Ghost-session *cleanup* — N/A. With wrapper-side pre-validation, kimi-cli is never invoked when a target is invalid, so no ghost directory gets created by P2's code paths. Pre-existing ghost directories (from past `/kimi:ask --resume <bogus>` invocations) are left in place; `session-empty` error code detects and refuses to resume them but does not delete.

### 1.3 Non-goals (explicit negative scope)

- **No interactive mode.** Claude Code slash commands are one-shot. `/kimi:continue` with no prompt → usage error, not "enter interactive session".
- **No state layer.** There is no "currently selected session" concept; every invocation resolves the target fresh. Avoids state-synchronization bugs.
- **No kimi.json writes.** The wrapper only *reads* `~/.kimi/kimi.json`. kimi-cli remains sole owner of write access.
- **No changes to `/kimi:rescue` or `task-resume-candidate`.** The rescue flow's `--resume-last` / `--fresh` semantics target recent rescue/task candidates, not cwd sessions; the two resume concepts coexist and stay decoupled.

**Design properties:**
- **Documented outcomes** (`kimi-json-missing` / `kimi-json-malformed` / `no-work-dir` / `no-last-session` / `invalid-uuid` / `session-not-found` / `session-empty` plus `ok:true`) are explicit `{ok, reason}` objects. Implementation uses internal try/catch so these never throw.
- **Unexpected FS errors** (`EACCES`, `EPERM`, `ENOTDIR`, `ELOOP`, dangling symlink dereference, etc.) are caught at the outer boundary and returned as `{ok:false, reason:'fs-error', detail}` — **not** thrown up the call stack. This preserves the "consumer always gets a structured outcome" contract without silently swallowing unexpected state.
- No I/O beyond fs reads. No directory writes, no process invocations.
- No `homedir()` captured into module-level constants — `kimiJsonPath()` and `sessionsDir()` re-read `$HOME` on each call, so tests that swap `process.env.HOME` after import still work.
- Consumer (`kimi-companion.mjs`) calls `mapSessionReason(r.reason, ctx)` on non-ok outcomes, writes the returned `errorResult.error` string to stderr, and calls `process.exit(errorResult.status)`.

```js
    return;
  }

  const realCwd = resolveRealCwd();
  const r1 = resolveContinueTarget(realCwd);
  if (!r1.ok) {
    const er = mapSessionReason(r1.reason, { realCwd });
    process.stderr.write(er.error + '\n');
    process.exit(er.status);
    return;
  }

  const r2 = validateResumeTarget(realCwd, r1.sessionId);
  if (!r2.ok) {
    const er = mapSessionReason(r2.reason, { realCwd, sessionId: r1.sessionId });
    process.stderr.write(er.error + '\n');
    process.exit(er.status);
    return;
  }

  const result = await callKimi({ prompt, resumeSessionId: r1.sessionId });
  hookTimingForResult(result);
  return result;
}

case 'resume': {
  const [sessionId, ...rest] = args;
  if (!sessionId) {
    process.stderr.write('Error: /kimi:resume requires <sessionId> <prompt>.\n');
    process.exit(2);
    return;
  }
  const prompt = rest.join(' ').trim();
  if (!prompt) {
    process.stderr.write('Error: /kimi:resume requires a <prompt> after the sessionId.\n');
    process.exit(2);
    return;
  }

  const realCwd = resolveRealCwd();
  const r = validateResumeTarget(realCwd, sessionId);
  if (!r.ok) {
    const er = mapSessionReason(r.reason, { realCwd, sessionId });
    process.stderr.write(er.error + '\n');
    process.exit(er.status);
    return;
  }

  const result = await callKimi({ prompt, resumeSessionId: sessionId });
  hookTimingForResult(result);
  return result;
}
```

**`ask` branch deletion:**

Inside the existing `case 'ask':` parser loop (`kimi-companion.mjs:253-257,949`), remove the arm that reads `--resume <value>` and the `resumeSessionId` pass-through to `callKimi`. After removal, an invocation like `/kimi:ask --resume X "hi"` naturally falls through to the "unknown flag" exit-2 path already implemented in the companion's option parser. Add one test case (in `tests/commands-p2.test.mjs`) to pin this regression.

**`mapSessionReason(reason, ctx)` helper** lives in `lib/sessions.mjs` (per §N1). Returns an `errorResult({kind, error, status, detail})` object ready for the stderr-write + exit pattern shown above. Per-reason template strings verbatim from §6.1.

Note: `callKimi` uses the parameter name `resumeSessionId` (see `lib/kimi.mjs:399,407,476,505,634,655,990`), not `resumeId`. Passing an unknown property silently ignores it, so correctness depends on using the exact name.

## 5. Data flow (end-to-end)

### 5.1 `/kimi:continue <prompt>` happy path

```
User types /kimi:continue <prompt>
  ↓
Claude Code invokes commands/continue.md's Bash line:
  KIMI_COMPANION_CALLER=claude node kimi-companion.mjs continue "$ARGUMENTS"
  ↓
companion main() dispatches → case 'continue':
  ↓
(a) prompt = args.join(' ').trim()     — empty → stderr.write + process.exit(2)
(b) realCwd = resolveRealCwd()         — P3 lib/paths.mjs; realpath-normalized
(c) r1 = resolveContinueTarget(realCwd)
    reads ~/.kimi/kimi.json, returns { ok, sessionId, reason }
    ok=false → mapSessionReason → stderr.write(er.error) + process.exit(er.status)
(d) r2 = validateResumeTarget(realCwd, r1.sessionId)
    — double-checks disk consistency in case kimi.json is stale (e.g., user manually
      rm'd the session dir after kimi-cli recorded it as last)
    reads ~/.kimi/sessions/<md5(realCwd)>/<sessionId>/
    ok=false → mapSessionReason → stderr.write(er.error) + process.exit(er.status)
(e) result = await callKimi({ prompt, resumeSessionId: r1.sessionId })
    — existing callKimi path from lib/kimi.mjs:505; adds '-r <sessionId>' to kimi-cli args
    — prompt transport: callKimi internally chooses '-p <str>' for small prompts
      and '--input-format text + stdin pipe' for large prompts (threshold governed
      by lib/kimi.mjs#useStdinForPrompt at line 399). P2 does not alter this.
(f) hookTimingForResult(result)
    — P1 helper; flushes { timings, exitCode, signal } into timing store
(g) render stdout: response verbatim + footer '(session: <uuid> · model: <name> ...)'
(h) exit 0
```

### 5.2 `/kimi:resume <sessionId> <prompt>` happy path

Same as §5.1 but skips step (c); step (a) parses `<sessionId> <prompt>` as two tokens. `validateResumeTarget` then runs directly against the user-supplied `sessionId`. Step (e) still passes `resumeSessionId: <validated>` to `callKimi`.

### 5.3 Error path

Any failure in (a)/(c)/(d) → companion exits non-zero with stderr-line `Error: ...` matching §6 templates. No `kimi -r` invocation ever happens on an unvalidated target, which is the primary safety property.

### 5.4 Timing hook integration

Both branches call `hookTimingForResult(result)` at step (f). `result` from `callKimi` includes `{ timings, exitCode, signal }` per P1's P1 T9 instrumentation. Timing records are persisted before the companion returns, so `/kimi:timing --last` immediately reflects the resume call.

Hooking for error paths (where `callKimi` never ran): **no hook** — there is no LLM call to time. errorResult flows out directly.

## 6. Error handling

### 6.1 Literal error templates

**Path-exposure policy (§S1)**: user-facing strings use `<cwd-basename>` (last path component) instead of the absolute path, to reduce accidental leakage of private filesystem layout into shells/logs/screenshots. The full `<realCwd>` still lives in the `errorResult.detail` field for debugging.

| `reason` | exit | stderr message (literal, user-facing) | `detail` field | Trigger |
|---|---|---|---|---|
| `no-work-dir` | 1 | `Error: no prior kimi session for this directory (<cwd-basename>). Use /kimi:ask to start one.` | `{realCwd}` | `/kimi:continue` when `work_dirs[realCwd]` is absent from `~/.kimi/kimi.json` |
| `kimi-json-missing` | 1 | `Error: no prior kimi session for this directory (<cwd-basename>). Use /kimi:ask to start one.` | `{realCwd}` | `/kimi:continue` when `~/.kimi/kimi.json` itself does not exist |
| `no-last-session` | 1 | `Error: no prior kimi session for this directory (<cwd-basename>). Use /kimi:ask to start one.` | `{realCwd}` | `/kimi:continue` when `work_dirs[realCwd]` exists but `last_session_id` is missing/empty |
| `kimi-json-malformed` | 1 | `Error: ~/.kimi/kimi.json is malformed; cannot resolve last session.` | `{parseError}` | `/kimi:continue` when JSON parse fails (treat trailing garbage as malformed — strict `JSON.parse`, do NOT fall back to permissive parsing) |
| `session-not-found` | 1 | `Error: session <uuid> not found for this directory (<cwd-basename>).` | `{realCwd, sessionId}` | `/kimi:continue` (stale kimi.json) OR `/kimi:resume` when the target UUID's directory does not exist under the current cwd's md5 — **including when it exists under a different cwd's md5** (cross-cwd refused, see §7.3) |
| `session-empty` | 1 | `Error: session <uuid> has no messages; refusing to resume ghost session.` | `{realCwd, sessionId, dirContents}` | Directory exists under current cwd's md5 but no populated message file — the hypothesized ghost-session state from probe v4 Q4.0/Q4.4 |
| `invalid-uuid` | 2 | `Error: invalid sessionId format; expected UUID.` | `{candidateId}` | `/kimi:resume` when first token does not match UUID regex (relaxed `/^[0-9a-f-]{36}$/i`) |
| `fs-error` | 1 | `Error: filesystem access failed — <errCode>. Check permissions on ~/.kimi/.` | `{errCode, errMessage, path}` | Any unexpected filesystem exception during session resolution/validation: EACCES, EPERM, ENOTDIR, ELOOP, dangling symlink dereference, uid mismatch, etc. |

**Exit code convention:** `1` = session-state or environment error (pre-validation refuses to invoke kimi-cli; user needs to fix state). `2` = usage/syntax error (bad argument shape; user needs to fix invocation). This matches the existing convention in `/kimi:rescue` for `2` = usage error (rescue.md line 61).

### 6.2 Declarative suggestions in command markdown

Following the ask.md pattern (declarative, no-question, single-line suggestions), the two new command markdowns add mapping in their "If companion exits non-zero" section:

- `no prior kimi session for this directory` → "Run `/kimi:ask <prompt>` to start a fresh session in this directory."
- `session <uuid> not found for this directory` → "Verify the sessionId or omit the flag to start fresh via `/kimi:ask`."
- `session <uuid> has no messages` → "The session is a ghost (no exchanges on disk); start a fresh session via `/kimi:ask`."
- `invalid sessionId format` → "Pass a valid UUID (8-4-4-4-12 hex). List your sessions by inspecting `~/.kimi/sessions/` if needed."
- `~/.kimi/kimi.json is malformed` → "Inspect the file by hand; recovery requires manual repair or starting fresh via `/kimi:ask`."
- `filesystem access failed` → "Check ~/.kimi/ directory and file permissions; ensure your current shell user owns the path."

### 6.3 Error-shape integration with P3 `errorResult()`

**Factual `errorResult()` contract** (from `plugins/kimi/scripts/lib/errors.mjs:16`):

```js
export function errorResult({ kind = null, error, status = null, stdout = "", detail = null } = {}) {
  // returns { ok: false, kind, error, status, stdout, detail }
}
```

`errorResult()` is a **plain object builder**. It does NOT:
- print to stderr or stdout on its own
- call `process.exit()`
- apply an exit-code convention

Routing the returned object (stdout JSON vs. stderr plain-text) and the actual `process.exit(<status>)` call are the calling branch's responsibility. Existing usage pattern examples live at `kimi-companion.mjs:427,481,536,607,653,664`.

P2 adds per-reason templates + a builder, both co-located in `lib/sessions.mjs` (§N1) so that `errors.mjs` stays provider-agnostic and does not learn session-domain semantics:

```js
// lib/sessions.mjs additions — templates verbatim from §6.1 table
const SESSION_ERROR_REASONS = {
  'no-work-dir':         { status: 1, template: 'Error: no prior kimi session for this directory ({cwdBase}). Use /kimi:ask to start one.' },
  'kimi-json-missing':   { status: 1, template: 'Error: no prior kimi session for this directory ({cwdBase}). Use /kimi:ask to start one.' },
  'no-last-session':     { status: 1, template: 'Error: no prior kimi session for this directory ({cwdBase}). Use /kimi:ask to start one.' },
  'kimi-json-malformed': { status: 1, template: 'Error: ~/.kimi/kimi.json is malformed; cannot resolve last session.' },
  'session-not-found':   { status: 1, template: 'Error: session {sessionId} not found for this directory ({cwdBase}).' },
  'session-empty':       { status: 1, template: 'Error: session {sessionId} has no messages; refusing to resume ghost session.' },
  'invalid-uuid':        { status: 2, template: 'Error: invalid sessionId format; expected UUID.' },
  'fs-error':            { status: 1, template: 'Error: filesystem access failed — {errCode}. Check permissions on ~/.kimi/.' },
};

import { basename } from 'node:path';
import { errorResult } from './errors.mjs';

export function mapSessionReason(reason, ctx = {}) {
  const spec = SESSION_ERROR_REASONS[reason];
  if (!spec) {
    return errorResult({ kind: 'sessions', error: `Error: internal — unknown session reason '${reason}'.`, status: 1, detail: { reason, ctx } });
  }
  const cwdBase = ctx.realCwd ? basename(ctx.realCwd) : '?';
  const filled = spec.template
    .replace('{cwdBase}', cwdBase)
    .replace('{sessionId}', ctx.sessionId || '?')
    .replace('{errCode}', ctx.errCode || '?');
  return errorResult({ kind: 'sessions', error: filled, status: spec.status, detail: { reason, ...ctx } });
}
```

**Consumer (kimi-companion.mjs)** pattern — same as existing branches:

```js
const er = mapSessionReason(r.reason, { realCwd, sessionId });
process.stderr.write(er.error + '\n');
process.exit(er.status);
```

All P2 stderr output goes through this mapping for consistency with kimi-companion.mjs existing error-handling convention. No changes to `lib/errors.mjs`.

## 7. Edge cases decision matrix

### 7.1 `/kimi:continue` — cwd never used with kimi

**State**: `work_dirs[realCwd]` is absent from `~/.kimi/kimi.json`, or `kimi.json` itself does not exist.
**Behavior**: Hard fail. Reason `no-work-dir` or `kimi-json-missing`. Exit 1. Same user-facing message either way.

### 7.2 `/kimi:continue` — stale kimi.json (session rm'd)

**State**: `work_dirs[realCwd].last_session_id = U`, but `~/.kimi/sessions/<md5(realCwd)>/<U>/` no longer exists (user manually removed).
**Behavior**: Hard fail at step (d) of §5.1. Reason `session-not-found`. Exit 1. The stale kimi.json is **not** auto-healed by P2 (would require a write); users can run any `/kimi:ask <prompt>` to create a fresh session which kimi-cli will record as new `last_session_id`.

### 7.3 `/kimi:resume <id>` — id exists under a different cwd

**State**: `~/.kimi/sessions/<md5(realCwd)>/<id>/` does not exist, but `~/.kimi/sessions/<md5(otherCwd)>/<id>/` does.
**Behavior**: Hard fail. Reason `session-not-found`. Cross-cwd resume is **explicitly disallowed** (see §1.2). The error message intentionally says "not found for `<realCwd>`" rather than leaking the fact that the session exists elsewhere — the information is not useful at the CLI level and leaking it invites "why doesn't this just work?" support questions.

### 7.4 `/kimi:resume <id>` — malformed UUID

**State**: First token does not match the relaxed UUID regex `/^[0-9a-f-]{36}$/i` (8-4-4-4-12 hex, any version digit).
**Behavior**: Hard fail. Reason `invalid-uuid`. Exit 2 (usage error). Declines to perform any disk I/O for invalid input. Regex is deliberately permissive (matches lib/kimi.mjs:63) to avoid blocking legitimate kimi-cli sessions if their UUIDs turn out not to be strict v4 — probe v4 Q4.5 pins the actual variant.

### 7.5 `/kimi:resume <id>` — empty-ghost session

**State**: Directory exists at `~/.kimi/sessions/<md5(normalizedCwd)>/<id>/` but no populated message file. The **exact predicate** for "populated" depends on probe v4 Q4.2 + Q4.4 (encoding — UTF-8 vs binary; file names — `messages.jsonl` / `session.yaml` / other; whether kimi-cli writes a placeholder file before first-message completion; distinction from real-session's first-turn state). Until probe v4 runs, the implementation uses a conservative rule: "no regular file with non-zero byte length and non-whitespace content" — probe v4 refines to a specific file-name or content-schema check.
**Behavior**: Hard fail. Reason `session-empty`. Specifically named so users know this is distinct from "not found" — they may have the right id, but the session is empty.

### 7.6 Concurrent modification of kimi.json

**State**: kimi-cli is running in another terminal and writes to `kimi.json` between our read and our validation.
**Behavior**: We do a fresh file read at step (c). If kimi.json is briefly truncated (partial write) or contains trailing garbage, strict `JSON.parse` throws — caught and returned as `kimi-json-malformed`; user can retry. We do **not** take a lock on kimi.json (kimi-cli owns writes; we're a reader). This is a known accept-the-race scenario: worst case is a spurious one-shot error visible to the user, retry fixes it. We also do **not** fall back to permissive parsing (§6.1 `kimi-json-malformed` row).

### 7.7 cwd path normalization — realpath vs verbatim

**Context**: probe v3 (`doc/probe/probe-results.json:47`) establishes that kimi-cli stores `work_dirs[].path` keys in **verbatim-absolute-normalized** form — absolute + dot-segment-removed but **NOT** symlink-resolved. Source: `kimi_cli/session.py::canonical()`. Probe 06 explicitly demonstrated that `md5(realpath(cwd))` does **not** match the stored hash when cwd is a symlink.

**Decision**: P2 wrapper calls `resolveRealCwd()` (P3 `lib/paths.mjs`) on **both** sides — the cwd passed to `resolveContinueTarget` + `validateResumeTarget` AND when the same cwd would be handed to kimi-cli (e.g., via `-w` if ever used by `callKimi`). This is the explicit workaround recommended by probe v3 results (`recovery: "always pass -w with the same form used for comparison (recommend fs.realpathSync(cwd) on both sides)"`).

**Why this works**: if our md5 and kimi-cli's md5 are both computed on the realpath-normalized string, they match. If both are computed on the verbatim string, they match. The only failure mode is mixing the two — which we prevent by centralizing normalization through `resolveRealCwd()`.

**Alternative considered and rejected**: using kimi-cli's verbatim `canonical()` logic (dot-segment removal only, no symlink resolution) wrapper-side. Rejected because: (a) we'd have to mirror a kimi-cli Python function in JS, introducing drift risk; (b) `fs.realpathSync()` is one well-known Node API while manual canonical() replication is ad-hoc; (c) realpath is idempotent so passing an already-realpath'd path through it is a no-op.

**Behavior**: Always call `resolveRealCwd()` before touching sessions. The symmetry property makes kimi.json lookup + sessions dir md5 match regardless of whether the user's shell cwd is a symlink or not.

### 7.8 Filesystem anomalies

Unusual filesystem state that should not crash the wrapper:

| State | Behavior |
|---|---|
| `process.env.HOME` unset | `homedir()` returns platform fallback (macOS `/var/root`); kimi-json-missing or no-work-dir will typically fire. Acceptable: the user is in a degenerate environment and should get a clear error, not a crash. |
| `~/.kimi/kimi.json` is a symlink | `readFileSync` follows symlinks; no special handling needed. If symlink is dangling → `ENOENT` → `kimi-json-missing`. |
| `~/.kimi/sessions/<md5>/` is a symlink | `statSync` / `readdirSync` follow symlinks. If dangling → `ELOOP` or `ENOENT` → returned as `fs-error`. |
| `~/.kimi/sessions/<md5>/<uuid>/` owned by different uid, not readable by current user | `EACCES` on `readdirSync` → returned as `fs-error` with detail. |
| `~/.kimi/kimi.json` contains trailing garbage after valid JSON | Strict `JSON.parse` rejects → `kimi-json-malformed` (§7.6). |
| `~/.kimi/sessions/<md5>/` does not exist at all (no prior kimi-cli use) | `readdirSync` throws `ENOENT` → caught at resolveContinueTarget boundary → if `resolveContinueTarget` called first, we never reach this; if `validateResumeTarget` called directly (e.g., `/kimi:resume X` in a fresh cwd), returns `session-not-found`. |
| `~/.kimi` itself does not exist | Same as `kimi-json-missing` (directory + file both absent). |

## 9. Breaking changes + migration

### 9.1 BREAKING: `/kimi:ask --resume <id>` removed

**What changes**: The `--resume <sessionId>` flag is deleted from `/kimi:ask` (both `argument-hint` frontmatter and the companion's option parser). Any existing invocation fails with "unknown flag" exit 2.

**Migration path**: `/kimi:ask --resume <uuid> <prompt>` → `/kimi:resume <uuid> <prompt>`. Exact same underlying behavior — `callKimi` is still invoked with `resumeSessionId: <uuid>` — plus the new validation guarantee.

**Affected users**: anyone who memorized the `/kimi:ask --resume` shape from v0.1. Spec-level expectation: low impact (sibling plugins never had this flag — see §0.3 scan paths; kimi docs have not heavily promoted it).

### 9.2 BREAKING: ghost-session path hardened

**What changes**: invocations that previously appeared to succeed with bogus UUIDs (hypothesized ghost-session creation per §0.2, pending probe v4 Q4.0 confirmation) now hard-fail with `session-not-found` or `session-empty`.

**Migration path**: verify UUIDs; use `/kimi:continue` for "keep going" intent, `/kimi:resume` only when you have the right id.

**Affected users**: anyone whose existing scripts pass UUIDs that silently drift to ghost sessions. Failure mode is loud and safe (no silent data loss; user sees clear error). If probe v4 Q4.0 falsifies the ghost hypothesis, this BREAKING note still applies descriptively — the validation still rejects invalid UUIDs, just without the dramatic framing.

### 9.3 PR-split trade-off (S4)

**Decision**: new commands + `/kimi:ask --resume` removal ship in the **same PR** (user-selected A1 during brainstorm).

**Arguments for the same-PR choice (chosen)**:
- Sibling-backport clarity: the entire "resume-UX overhaul" lands as one unit; future gemini/minimax/qwen backports follow a single diff.
- Single CHANGELOG entry telling the full story instead of two partial stories.
- Test coverage for the removal lives in the same `tests/commands-p2.test.mjs` as the new commands, avoiding a standalone test file for a one-line parser removal.

**Arguments against (considered and rejected)**:
- **Attribution risk on regression**: if a bug surfaces post-merge, bisect can attribute it to "the P2 commit" but isolating cause between "new command path" vs. "ask.md option-parser change" takes an extra read. Counter: git-blame on the specific line is sufficient; this is a minor forensic inconvenience, not a blocker.
- **Review surface**: a same-PR change is larger. Counter: the ask.md delta is ~5 lines total (frontmatter hint + option parser arm + pass-through), dwarfed by the new-command scaffolding.

**Conclusion**: A1 (same PR) remains the right call. Documented here so future spec reviewers or a post-merge incident investigator can see the trade-off was considered, not overlooked.

## 12. Open questions (for 6-way review)

### 12.1 Closed since v1 (kept for audit trail)

- **[CLOSED in v1.1] cwd path normalization — realpath vs verbatim**: decided per §7.7 revision. Wrapper normalizes both sides via `resolveRealCwd()`; this is the probe-v3-recommended consistency workaround, NOT a claim that kimi-cli uses realpath (kimi-cli uses verbatim per probe-results.json:47). Rationale in §7.7 "Alternative considered and rejected".
- **[CLOSED in v1.1] `mapSessionReason` placement**: decided per §N1 revision. Lives in `lib/sessions.mjs` (reason-producer owns mapping). Rationale: `lib/errors.mjs` is provider-agnostic and should not learn session-domain concepts. The cohesion argument for co-locating with errors.mjs is weaker than the locality argument for co-locating with the function that produces the reasons.
- **[CLOSED in v1.1] ask.md removal timing (same PR vs. split)**: decided per §9.3 revision. Same PR (A1). Rationale documented in full there.

### 12.2 Still open (6-way reviewers, please weigh in)

1. **UUID regex strictness**: current plan uses relaxed `/^[0-9a-f-]{36}$/i` (matching lib/kimi.mjs:63 convention). Probe v4 Q4.5 pins the kimi-cli UUID variant. If kimi-cli happens to generate nil UUIDs (`00000000-...`) for some edge case, a relaxed regex is necessary; if it always generates strict v4, a stricter regex gives a slightly better error message for malformed input. Low-stakes — leaning permissive stays current.
2. **kimi.json concurrent-write window**: §7.6 accepts a race where a brief read during kimi-cli write may return `kimi-json-malformed`. Should we retry-once with a 50ms delay before surfacing the error? Tradeoff: extra latency on happy path (milliseconds) vs. rare spurious error (seconds of user confusion). Weak preference: no retry — a retry masks the rare real case where kimi.json is persistently malformed, and users can manually retry.
3. **Ghost-session counter-proposal**: should P2 additionally *delete* detected ghost sessions (`rm -rf <ghost-dir>`) as a courtesy cleanup, or strictly stay read-only? Current spec: strictly read-only (refuse + do not delete). Deleting is tempting but conflicts with "kimi-cli is sole owner of sessions" boundary. Probe v4 Q4.0/Q4.4 may reveal ghost dirs are always distinguishable; if so, a hypothetical `/kimi:cleanup-ghosts` could land in a later phase.
4. **`--model` flag parity**: if a user wants to resume session X but try a different model, current spec says defer. Is deferring OK, or do users request this parity with `/kimi:ask --model`? v0.2 evidence: no one asked yet. YAGNI stays.
5. **fs-error granularity**: current spec lumps EACCES / EPERM / ENOTDIR / ELOOP / dangling symlink all under single `fs-error` reason. Should any of these split out (e.g., a dedicated `permission-denied` reason with distinct user message)? Low-value — permissions on `~/.kimi/` are uniformly rare.
6. **Probe v4 falsification plan**: if Q4.0 shows kimi-cli does NOT create ghost sessions (or the ghost is somehow not triggered in `--print` mode the way source suggests), how much of the spec stays? Current answer: all of it — wrapper validation is still defensible as input-sanity hardening. But the dramatic "BREAKING: ghost-session hardened" framing in §9.2 would soften. Reviewers: if you've read kimi-cli source recently, flag whether the PR #1716 path in `cli/__init__.py::_handle_resume_flag` (or similar) might have been backported to `--print` in a newer kimi-cli version we haven't probed.
