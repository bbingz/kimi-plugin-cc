## 0. Context / Motivation

### 0.1 Why /kimi:continue + /kimi:resume

Today a Kimi user who wants to resume a prior conversation has two unattractive options:

1. `/kimi:ask --resume <uuid> <prompt>` — works, but (a) requires the user to already know the UUID, (b) **has a silent ghost-session bug** (probe v4 Q5): passing a bogus UUID creates an empty session at that UUID and treats it as a fresh start, without telling the user.
2. `/kimi:rescue --resume-last` — works, but `--resume-last` means "resume the most recent *rescue/task* candidate" (see `kimi-companion.mjs task-resume-candidate`), **not** "resume the most recent cwd session". Mismatched mental model.

There is no ergonomic way to say "keep working on what we were just doing in this repo." P2 fills that gap with two new slash commands:

- `/kimi:continue <prompt>` — resume the most recent session for the current working directory.
- `/kimi:resume <sessionId> <prompt>` — resume a specific session by UUID (scoped to the current cwd).

### 0.2 Hard constraint from probe v4 Q5

`kimi --print -r <bogus-uuid>` **silently creates a ghost session** at that UUID rather than rejecting. The interactive-mode `-r` validation raise introduced in kimi-cli PR #1716 does not fire in `--print` mode. Any P2 command that invokes `kimi -r` without wrapper-side pre-validation inherits this bug.

**Therefore every P2 resume entry point MUST validate the resume target before invoking kimi-cli.** This single constraint shapes the entire design.

### 0.3 Sibling-series significance

`/kimi:continue` and `/kimi:resume` are the **first** user-facing resume commands in the kimi-plugin-cc template series (scan of 2026-04-22: gemini/minimax/qwen plugins have no such commands; `/qwen:ask` even explicitly declares itself as "always a new conversation"). P2 sets the template that future sibling backports (gemini, minimax, qwen, eventual doubao) will mirror.

The sibling precedent is also why P2 removes the `/kimi:ask --resume` flag (see §9): three of three siblings have `ask` as "new-conversation-only"; keeping `/kimi:ask --resume` as a kimi-only special case would create permanent "why does kimi have this but nobody else?" documentation drag.

### 1.2 Out of scope (YAGNI)

- `/kimi:list-sessions` — deferred. Rationale: users who need a listing can `jq` `~/.kimi/kimi.json` or `ls ~/.kimi/sessions/<md5(cwd)>/`; no evidence yet that a first-class listing UX is required. Revisit after P2 ships.
- `--model <name>` flag on continue/resume — deferred. Resume inherits the session's original model (kimi-cli behavior); overriding mid-stream complicates validation. Revisit if users request it.
- `--stream` flag on continue/resume — deferred. Matches ask.md's rule: `--stream` is developer-only when invoking the companion directly; slash commands always go through the gated (non-stream) path.
- Cross-cwd resume — **explicitly disallowed** (see §7). Session-to-cwd is 1:1 via `md5(realCwd)` path; resuming a session that belongs to a different cwd would either fail silently in kimi-cli or create inconsistent state. Not worth the test matrix.
- Ghost-session *cleanup* — N/A. With wrapper-side pre-validation, kimi-cli is never invoked when a target is invalid, so no ghost directory gets created by P2's code paths. Pre-existing ghost directories (from past `/kimi:ask --resume <bogus>` invocations) are left in place; `session-empty` error code detects and refuses to resume them but does not delete.

### 1.3 Non-goals (explicit negative scope)

- **No interactive mode.** Claude Code slash commands are one-shot. `/kimi:continue` with no prompt → usage error, not "enter interactive session".
- **No state layer.** There is no "currently selected session" concept; every invocation resolves the target fresh. Avoids state-synchronization bugs.
- **No kimi.json writes.** The wrapper only *reads* `~/.kimi/kimi.json`. kimi-cli remains sole owner of write access.
- **No changes to `/kimi:rescue` or `task-resume-candidate`.** The rescue flow's `--resume-last` / `--fresh` semantics target recent rescue/task candidates, not cwd sessions; the two resume concepts coexist and stay decoupled.

## 4. Components

### 4.1 `/kimi:continue` (commands/continue.md)

```markdown
---
description: Continue the most recent Kimi session for the current cwd
argument-hint: '<prompt>'
allowed-tools: Bash(node:*)
---

Run with the companion-caller env so the companion knows this is a slash-command invocation:

```bash
KIMI_COMPANION_CALLER=claude node "${CLAUDE_PLUGIN_ROOT}/scripts/kimi-companion.mjs" continue "$ARGUMENTS"
```

The companion runs in text mode and produces:
- Line 1..N: Kimi's response verbatim
- Last line: footer `(session: <uuid> · model: <name> [· thinkBlocks: N])`

**If companion exits 0**: render stdout verbatim (same rules as `/kimi:ask`). **MUST NOT** prepend or append commentary.

**If companion exits non-zero**: present stderr `Error: ...` verbatim. Add one declarative suggestion per §6 template mapping. Do NOT retry automatically.

### Notes

- `<prompt>` is required. Empty `$ARGUMENTS` returns usage error (exit 2).
- Resumes the most recent session for the current working directory. Fails cleanly if no prior session exists (see §6 error templates).
```

### 4.2 `/kimi:resume` (commands/resume.md)

Same shape as `continue.md`, except:
- `argument-hint: '<sessionId> <prompt>'`
- `$ARGUMENTS` first token parsed as sessionId (UUID); rest is prompt.
- Usage errors: empty `$ARGUMENTS` → "usage: /kimi:resume <sessionId> <prompt>"; single-token `$ARGUMENTS` → "missing prompt".

### 4.3 `/kimi:ask` modifications (commands/ask.md)

- `argument-hint` frontmatter: change `[--model <model>] [--resume <sessionId>] <prompt>` → `[--model <model>] <prompt>`.
- `### Options` section: delete the `--resume <sessionId>` bullet.
- No other content changes.

### 4.4 `lib/sessions.mjs` public API

**Design properties:**
- Zero throws; all outcomes are explicit `{ok, reason}` objects.
- No I/O beyond fs reads. (No `stat` on kimi.json's mtime, no directory writes, no process invocations.)
- Consumer (`kimi-companion.mjs`) maps `reason` → errorResult + exit code.
- Tests can populate a fake `$HOME` via env override (`process.env.HOME` at test setup) without any monkey-patching.

### 4.5 `kimi-companion.mjs` changes

**New branches** (near existing `case 'ask':`):

```js
case 'continue': {
  const prompt = args.join(' ').trim();
  if (!prompt) return usageError(`/kimi:continue requires a <prompt>`);

  const realCwd = resolveRealCwd();
  const r1 = resolveContinueTarget(realCwd);
  if (!r1.ok) return errorResult(mapSessionReason(r1.reason, { realCwd }));

  const r2 = validateResumeTarget(realCwd, r1.sessionId);
  if (!r2.ok) return errorResult(mapSessionReason(r2.reason, { realCwd, sessionId: r1.sessionId }));

  const result = await callKimi({ prompt, resumeId: r1.sessionId, model: undefined });
  hookTimingForResult(result);
  return result;
}

case 'resume': {
  const [sessionId, ...rest] = args;
  if (!sessionId) return usageError(`/kimi:resume requires <sessionId> <prompt>`);
  const prompt = rest.join(' ').trim();
  if (!prompt) return usageError(`/kimi:resume requires a <prompt> after the sessionId`);

  const realCwd = resolveRealCwd();
  const r = validateResumeTarget(realCwd, sessionId);
  if (!r.ok) return errorResult(mapSessionReason(r.reason, { realCwd, sessionId }));

  const result = await callKimi({ prompt, resumeId: sessionId, model: undefined });
  hookTimingForResult(result);
  return result;
}
```

**`ask` branch deletion:**

Inside the existing `case 'ask':` parser loop, remove the arm that reads `--resume <value>`. After removal, an invocation like `/kimi:ask --resume X "hi"` naturally falls through to the "unknown flag" exit-2 path. Add one test case to prevent regression.

**`mapSessionReason(reason, ctx)` helper** (exported from `lib/errors.mjs`, new function): returns the appropriate errorResult object based on §6 template mapping.

## 5. Data flow (end-to-end)

### 5.1 `/kimi:continue <prompt>` happy path

```
User types /kimi:continue <prompt>
  ↓
Claude Code invokes commands/continue.md's Bash line:
  KIMI_COMPANION_CALLER=claude node kimi-companion.mjs continue "$ARGUMENTS"
  ↓
companion main() dispatches → case 'continue':
  ↓
(a) prompt = parseArgs(args)           — fail → usageError exit 2
(b) realCwd = resolveRealCwd()         — P3 lib/paths.mjs; stable across symlinks
(c) r1 = resolveContinueTarget(realCwd)
    reads ~/.kimi/kimi.json, returns { ok, sessionId, reason }
    ok=false → errorResult + exit 1 (§6 mapping)
(d) r2 = validateResumeTarget(realCwd, r1.sessionId)
    — double-checks disk consistency in case kimi.json is stale (e.g., user manually
      rm'd the session dir after kimi-cli recorded it as last)
    reads ~/.kimi/sessions/<md5(realCwd)>/<sessionId>/
    ok=false → errorResult + exit 1 (§6 mapping)
(e) result = await callKimi({ prompt, resumeId: r1.sessionId })
    — existing callKimi path from lib/kimi.mjs; adds -r <sessionId> to kimi-cli args
    — stdin format: --input-format text + piped stdin (per user's kimi-cli stdin note)
(f) hookTimingForResult(result)
    — P1 helper; flushes { timings, exitCode, signal } into timing store
(g) render stdout: response verbatim + footer '(session: <uuid> · model: <name> ...)'
(h) exit 0
```

### 5.2 `/kimi:resume <sessionId> <prompt>` happy path

Same as §5.1 but skips step (c); step (a) parses `<sessionId> <prompt>` as two tokens.

### 5.3 Error path

Any failure in (a)/(c)/(d) → companion exits non-zero with stderr-line `Error: ...` matching §6 templates. No `kimi -r` invocation ever happens on an unvalidated target, which is the primary safety property.

### 5.4 Timing hook integration

Both branches call `hookTimingForResult(result)` at step (f). `result` from `callKimi` includes `{ timings, exitCode, signal }` per P1's P1 T9 instrumentation. Timing records are persisted before the companion returns, so `/kimi:timing --last` immediately reflects the resume call.

Hooking for error paths (where `callKimi` never ran): **no hook** — there is no LLM call to time. errorResult flows out directly.

## 6. Error handling

### 6.1 Literal error templates

| `reason` | exit | stderr message (literal) | Trigger |
|---|---|---|---|
| `no-work-dir` | 1 | `Error: no prior kimi session for <realCwd>. Use /kimi:ask to start one.` | `/kimi:continue` when `work_dirs[realCwd]` is absent from `~/.kimi/kimi.json` |
| `kimi-json-missing` | 1 | `Error: no prior kimi session for <realCwd>. Use /kimi:ask to start one.` | `/kimi:continue` when `~/.kimi/kimi.json` itself does not exist |
| `no-last-session` | 1 | `Error: no prior kimi session for <realCwd>. Use /kimi:ask to start one.` | `/kimi:continue` when `work_dirs[realCwd]` exists but `last_session_id` is missing/empty |
| `kimi-json-malformed` | 1 | `Error: ~/.kimi/kimi.json is malformed; cannot resolve last session.` | `/kimi:continue` when JSON parse fails |
| `session-not-found` | 1 | `Error: session <uuid> not found for <realCwd>.` | `/kimi:continue` (stale kimi.json) OR `/kimi:resume` when the target UUID's directory does not exist under the current cwd's md5 — **including when it exists under a different cwd's md5** (cross-cwd refused, see §7.3) |
| `session-empty` | 1 | `Error: session <uuid> has no messages; refusing to resume ghost session.` | Directory exists under current cwd's md5 but no serialized-message file (the ghost-session state from probe Q5) |
| `invalid-uuid` | 2 | `Error: invalid sessionId format; expected UUID.` | `/kimi:resume` when first token does not match UUID regex |

**Exit code convention:** `1` = session-state error (pre-validation refuses to invoke kimi-cli). `2` = usage/syntax error (bad argument shape). This matches the existing convention in `/kimi:rescue` for `2` = usage error (see rescue.md line 61).

### 6.2 Declarative suggestions in command markdown

Following the ask.md pattern (declarative, no-question, single-line suggestions), the two new command markdowns add mapping in their "If companion exits non-zero" section:

- `no prior kimi session` → "Run `/kimi:ask <prompt>` to start a fresh session in this directory."
- `session <uuid> not found` → "Verify the sessionId or omit the flag to start fresh via `/kimi:ask`."
- `session <uuid> has no messages` → "The session is a ghost (no exchanges on disk); start a fresh session via `/kimi:ask`."
- `invalid sessionId format` → "Pass a valid UUID (8-4-4-4-12 hex). List your sessions by inspecting `~/.kimi/sessions/` if needed."
- `~/.kimi/kimi.json is malformed` → "Inspect the file by hand; recovery requires manual repair or starting fresh via `/kimi:ask`."

## 7. Edge cases decision matrix

### 7.1 `/kimi:continue` — cwd never used with kimi

**State**: `work_dirs[realCwd]` is absent from `~/.kimi/kimi.json`, or `kimi.json` itself does not exist.
**Behavior**: Hard fail. Reason `no-work-dir` or `kimi-json-missing`. Exit 1. Same user-facing message either way.

### 7.2 `/kimi:continue` — stale kimi.json (session rm'd)

**State**: `work_dirs[realCwd].last_session_id = U`, but `~/.kimi/sessions/<md5(realCwd)>/<U>/` no longer exists (user manually removed).
**Behavior**: Hard fail at step (d) of §5.1. Reason `session-not-found`. Exit 1. The stale kimi.json is **not** auto-healed by P2 (would require a write); users can run any `/kimi:ask <prompt>` to create a fresh session which kimi-cli will record as new `last_session_id`.

### 7.3 `/kimi:resume <id>` — id exists under a different cwd

**State**: `~/.kimi/sessions/<md5(realCwd)>/<id>/` does not exist, but `~/.kimi/sessions/<md5(otherCwd)>/<id>/` does.
**Behavior**: Hard fail. Reason `session-not-found`. Cross-cwd resume is **explicitly disallowed** (see §1.2). The error message intentionally says "not found for `<realCwd>`" rather than leaking the fact that the session exists elsewhere — the information is not useful at the CLI level and leaking it invites "why doesn't this just work?" support questions.

### 7.4 `/kimi:resume <id>` — malformed UUID

**State**: First token does not match UUID v4 regex.
**Behavior**: Hard fail. Reason `invalid-uuid`. Exit 2 (usage error). Declines to perform disk I/O for invalid input.

### 7.5 `/kimi:resume <id>` — empty-ghost session

**State**: Directory exists at `~/.kimi/sessions/<md5(realCwd)>/<id>/` but contains no serialized-message file (probe v5 Q5.2 + Q5.4 determine the exact detection rule).
**Behavior**: Hard fail. Reason `session-empty`. Specifically named so users know this is distinct from "not found" — they may have the right id, but the session is empty.

### 7.6 Concurrent modification of kimi.json

**State**: kimi-cli is running in another terminal and writes to `kimi.json` between our read and our validation.
**Behavior**: We do a fresh file read at step (c). If kimi.json is briefly truncated (partial write), step (c) may return `kimi-json-malformed`; user can retry. We do **not** take a lock on kimi.json (kimi-cli owns writes; we're a reader). This is a known accept-the-race scenario: worst case is a spurious one-shot error visible to the user, retry fixes it.

### 7.7 `realpath` vs non-realpath cwd

**State**: User's cwd is a symlink, e.g., `/Users/bing/project-link → /Users/bing/actual/project`. kimi-cli keys `work_dirs` by realpath; if we keyed by non-realpath we'd miss.
**Behavior**: Always call `resolveRealCwd()` (P3 `lib/paths.mjs`) before touching sessions. Guaranteed consistency with kimi-cli's own logic. `md5CwdPath(realCwd)` uses the same realpath'd string.

## 9. Breaking changes + migration

### 9.1 BREAKING: `/kimi:ask --resume <id>` removed

**What changes**: The `--resume <sessionId>` flag is deleted from `/kimi:ask`. Any existing invocation fails with "unknown flag" exit 2.

**Migration path**: `/kimi:ask --resume <uuid> <prompt>` → `/kimi:resume <uuid> <prompt>`. Exact same underlying behavior (resume session + send prompt + print), plus the new validation guarantee.

**Affected users**: anyone who memorized the `/kimi:ask --resume` shape from v0.1. Spec-level expectation: low impact (sibling plugins never had this flag; kimi docs have not heavily promoted it).

### 9.2 BREAKING: ghost-session path hardened

**What changes**: invocations that used to silently succeed (creating a ghost session for a bogus UUID) now hard-fail.

**Migration path**: verify UUIDs; use `/kimi:continue` for "keep going" intent, `/kimi:resume` only when you have the right id.

**Affected users**: anyone whose existing scripts pass UUIDs that silently drift to ghost sessions. Failure mode is loud and safe (no silent data loss; user sees clear error).

### 9.3 CHANGELOG entry shape

Target file: `plugins/kimi/CHANGELOG.md` (reverse-chronological, top of next version block).

Example (to be finalized at merge):

```md
## Unreleased

### BREAKING
- `/kimi:ask --resume <sessionId>` flag removed. Use `/kimi:resume <sessionId> <prompt>` instead.
- Invalid resume targets now hard-fail instead of silently creating ghost sessions (fixes probe v4 Q5).

### Added
- `/kimi:continue <prompt>` — resume most recent session for current cwd.
- `/kimi:resume <sessionId> <prompt>` — resume specific session by UUID (cwd-scoped).
- `lib/sessions.mjs` — session resolution + validation (pure functions, reused by all resume entry points).

P2 becomes the **reference implementation** for "add resume commands to a CLI-wrapper plugin." Future siblings should diff their P2 PR against kimi's for structure while adapting specifics. Document this role explicitly in `sibling-backport-checklist.md`.

## 12. Open questions (for 6-way review)

1. **UUID regex strictness**: current plan uses UUID v4 shape (8-4-4-4-12 hex). Does kimi-cli actually generate v4, or mixed v1/v4/nil UUIDs? If mixed, a relaxed regex may be needed. (Probe v5 Q5.2 inspects the session dir name, which is the UUID; this surfaces the kimi-cli UUID variant.)
2. **kimi.json concurrent-write window**: §7.6 accepts a race where a brief read during kimi-cli write may return `kimi-json-malformed`. Should we retry-once with a 50ms delay before surfacing the error? Tradeoff: extra latency on happy path vs. rare spurious error.
3. **Ghost-session counter-proposal**: should P2 additionally *delete* detected ghost sessions (`rm -rf <ghost-dir>`) as a courtesy cleanup, or strictly stay read-only? Current spec: strictly read-only (refuse + do not delete). Deleting is tempting but conflicts with "kimi-cli is sole owner of sessions" boundary.
4. **`--model` flag parity**: if a user wants to resume session X but try a different model, current spec says defer. Is deferring OK, or do users request this parity with `/kimi:ask --model`? (v0.2 evidence: no one asked yet.)
5. **`mapSessionReason` placement**: the spec puts it in `lib/errors.mjs`. Alternative: keep it inside `lib/sessions.mjs`. Tradeoff: errors.mjs is the canonical error registry (cohesive), but sessions.mjs owns the reasons it produces (locality). Current choice: errors.mjs for canonical cohesion.
6. **ask.md removal timing**: should the `--resume` removal land in the same PR as the new commands, or ship first as a standalone deprecate/remove cycle? Current spec: same PR (P2 is the single "resume-UX overhaul" unit). Opposing argument: a separate ask.md PR would isolate the breaking change.
