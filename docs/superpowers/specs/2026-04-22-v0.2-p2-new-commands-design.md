## 0. Context / Motivation

### 0.1 Why /kimi:continue + /kimi:resume

Today a Kimi user who wants to resume a prior conversation has two unattractive options:

1. `/kimi:ask --resume <uuid> <prompt>` — works, but (a) requires the user to already know the UUID, (b) **is hypothesized to have a silent ghost-session bug** (source-read from kimi-cli PR #1716; pending probe v4 verification — see §0.2): passing a bogus UUID may silently create an empty session at that UUID and treat it as a fresh start, without telling the user.
2. `/kimi:rescue --resume-last` — works, but `--resume-last` means "resume the most recent *rescue/task* candidate" (see `kimi-companion.mjs task-resume-candidate`), **not** "resume the most recent cwd session". Mismatched mental model.

There is no ergonomic way to say "keep working on what we were just doing in this repo." P2 fills that gap with two new slash commands:

- `/kimi:continue <prompt>` — resume the most recent session for the current working directory.
- `/kimi:resume <sessionId> <prompt>` — resume a specific session by UUID (scoped to the current cwd).

### 0.2 Hard constraint — probe-confirmed + source-read (v2)

**Verdict (probe v4 Q4.0, commit `7529f1e`)**: `kimi --print -r <fabricated-uuid>` silently creates a ghost session at that UUID. Confirmed on kimi-cli 1.37.0 with: exit 0, full session directory materialized (`context.jsonl` + `state.json` + `wire.jsonl`), and `~/.kimi/kimi.json` `last_session_id` updated to the fabricated UUID. No stderr warning. User is given the same resume-hint they would get for a real session.

**Corrected scope claim** (v2, per Kimi source-read): the v1/v1.1/v1.2 spec attributed this to "PR #1716 interactive-mode validation raise not firing in --print mode". That attribution was WRONG. Kimi reviewer (2026-04-23, 6-way round 1) read `kimi_cli/cli/__init__.py:546-553` and confirmed: when `Session.find(work_dir, session_id)` returns None, the code unconditionally calls `Session.create(work_dir, session_id)` — there is no mode-specific validation branch. PR #1716 is not in the source tree. **The ghost-session behavior affects ALL modes (interactive, print, ACP, wire) equally.** P2's wrapper-side validation is therefore the only defense in any mode the plugin uses (P2 is print-only, but sibling plugins may use interactive via a different path; the same risk applies).

**Therefore every P2 resume entry point MUST validate the resume target before invoking kimi-cli.** This shapes the error matrix (§6/§7) and the `lib/sessions.mjs` API (§4.4).

**Cross-cwd extension (probe v4 Q4.3)**: the same ghost behavior triggers when resuming a UUID that belongs to a different cwd — kimi creates a duplicate session directory under the current cwd's md5 path. §7.3 wrapper-side cwd-scoped refusal is the only safe way to prevent accidental session forking.

**Concurrent-write upgrade** (v2, per MiniMax finding): the race scenario in §7.6 was originally framed as "spurious malformed JSON" — that framing understated severity. The actual worst-case race is: wrapper reads kimi.json and sees `last_session_id = U1`, kimi-cli in another terminal writes `last_session_id = U2`, wrapper validates U1 (still exists on disk as a real session), wrapper invokes `kimi -r U1`, kimi resumes U1 instead of the user's genuinely-recent U2. The user experiences a **wrong-session resume** with zero error indication. Mitigation in v2: §7.6 now acknowledges this as "observational; no deterministic test; accepted because the window is narrow and the fix (file-locking kimi.json) would conflict with kimi-cli's write-side ownership". The mitigation is documented so future incident investigators understand the known-limitation.

### 0.3 Sibling-series significance

`/kimi:continue` and `/kimi:resume` are the **first** user-facing resume commands in the kimi-plugin-cc template series. Scan conducted 2026-04-22 at these literal paths:
- `/Users/bing/-Code-/gemini-plugin-cc/plugins/gemini/commands/ask.md` — no `--resume` or `--session` flag
- `/Users/bing/-Code-/minimax-plugin-cc/plugins/minimax/commands/ask.md` — no `--resume` or `--session` flag
- `/Users/bing/-Code-/qwen-plugin-cc/plugins/qwen/commands/ask.md` — explicitly declares `/qwen:ask` is "always a new conversation; resume belongs to /qwen:rescue --resume-last"

Only kimi's `/kimi:ask` uniquely has `--resume <sessionId>` today (which P2 removes — see §9). P2 sets the template that future sibling backports (gemini, minimax, qwen, eventual doubao) will mirror.

The sibling precedent is also why P2 removes the `/kimi:ask --resume` flag (see §9): three of three siblings have `ask` as "new-conversation-only"; keeping `/kimi:ask --resume` as a kimi-only special case would create permanent "why does kimi have this but nobody else?" documentation drag.

### 1.2 Out of scope (YAGNI)

- `/kimi:list-sessions` — deferred. Rationale: users who need a listing can `jq` `~/.kimi/kimi.json` or `ls ~/.kimi/sessions/<md5(cwd)>/`; no evidence yet that a first-class listing UX is required. Revisit after P2 ships.
- `--model <name>` flag on continue/resume — deferred. Resume inherits the session's original model (kimi-cli behavior); overriding mid-stream complicates validation. Revisit if users request it.
- `--stream` flag on continue/resume — deferred. Matches ask.md's rule: `--stream` is developer-only when invoking the companion directly; slash commands always go through the gated (non-stream) path.
- Cross-cwd resume — **explicitly disallowed** (see §7). Session-to-cwd is 1:1 via `md5(realCwd)` path; resuming a session that belongs to a different cwd would either fail silently in kimi-cli or create inconsistent state. Not worth the test matrix.
- Ghost-session *cleanup* — N/A. With wrapper-side pre-validation, kimi-cli is never invoked when a target is invalid, so no ghost directory gets created by P2's code paths. Pre-existing ghost directories (from past `/kimi:ask --resume <bogus>` invocations) are left in place; `session-empty` error code detects and refuses to resume them but does not delete.

### 1.3 Non-goals (explicit negative scope)

- **No interactive mode.** Claude Code slash commands are one-shot. `/kimi:continue` with no prompt → usage error, not "enter interactive session".
- **No state layer.** There is no "currently selected session" concept; every invocation resolves the target fresh. Avoids state-synchronization bugs.
- **No kimi.json writes.** The wrapper only *reads* `~/.kimi/kimi.json`. kimi-cli remains sole owner of write access.
- **No changes to `/kimi:rescue` or `task-resume-candidate`.** The rescue flow's `--resume-last` / `--fresh` semantics target recent rescue/task candidates, not cwd sessions; the two resume concepts coexist and stay decoupled.

### 2.4 Assumptions — probe-confirmed (v1.2)

Probe v4 executed 2026-04-23 against kimi-cli 1.37.0. All three v1.1 assumptions updated with probe-confirmed findings:

- **Assumption A1 (CONFIRMED with schema correction)**: `~/.kimi/kimi.json` has only a top-level `work_dirs` key. Critical: `work_dirs` is a **LIST** (array of `{path, kaos, last_session_id}` objects), **NOT** an object keyed by path as v1.1 originally wrote. `path` field is **realpath-resolved** on kimi 1.37 (e.g., `/private/tmp/...` not `/tmp/...` on macOS) — this differs from probe v3's 1.36 finding of verbatim paths. Spec §4.4 API uses `work_dirs.find(w => w.path === normalizedCwd)` accordingly. Confidence: **high (probe v4 Q4.1)**.
- **Assumption A2 (CONFIRMED)**: `~/.kimi/sessions/<md5(realpath(cwd))>/<uuid>/` has exactly three files: `context.jsonl` (messages, UTF-8 JSONL, ≥17 KB for a trivial 1-turn exchange), `state.json` (metadata, ~400 B, UTF-8 JSON object with keys `[version, approval, additional_dirs, custom_title, title_generated, title_generate_attempts, plan_mode, plan_session_id, plan_slug, wire_mtime, archived, archived_at, auto_archive_exempt, todos]`), `wire.jsonl` (wire events, ~1 KB). "Populated" predicate for `validateResumeTarget`: `existsSync(dir + '/context.jsonl') && statSync(dir + '/context.jsonl').size > 0`. Confidence: **high (probe v4 Q4.2)**.
- **Assumption A3 (REFINED — post-hoc indistinguishable)**: Once kimi has run with a given UUID (whether legit or ghost), the resulting session directory is structurally identical to a real session. There is no post-hoc heuristic to detect "this was a ghost". Therefore the only defense is **pre-call validation** (already the design): we check the dir state BEFORE invoking `kimi -r`. `session-empty` reason code catches the narrow case of "dir exists but context.jsonl absent / zero-byte" — most likely a user-created placeholder or a kimi crash between mkdir and file-write (not observed in probe; theoretical edge). Confidence: **high (probe v4 Q4.4)**.

No further probe is strictly required before plan phase. A probe v4.1 extension for `kimi -C` ghost behavior would be needed if §12.2 Q6 is resolved toward "use `kimi -C` natively" — current preferred path (explicit `resolveContinueTarget → kimi -r <uuid>`) does not need it.

## 4. Components

### 4.1 `/kimi:continue` (commands/continue.md)

```markdown
---
description: Continue the most recent Kimi session for the current cwd
argument-hint: '<prompt>'
allowed-tools: Bash(node:*)
---

Run with the companion-caller env so the companion knows this is a slash-command invocation:

```bash
KIMI_COMPANION_CALLER=claude node "${CLAUDE_PLUGIN_ROOT}/scripts/kimi-companion.mjs" continue "$ARGUMENTS"
```

The companion runs in text mode and produces:
- Line 1..N: Kimi's response verbatim
- Last line: footer `(session: <uuid> · model: <name> [· thinkBlocks: N])`

**If companion exits 0**: render stdout verbatim (same rules as `/kimi:ask`). **MUST NOT** prepend or append commentary.

**If companion exits non-zero**: present stderr `Error: ...` verbatim. Add one declarative suggestion per §6 template mapping. Do NOT retry automatically.

### Notes

- `<prompt>` is required. Empty `$ARGUMENTS` returns usage error (exit 2).
- Resumes the most recent session for the current working directory. Fails cleanly if no prior session exists (see §6 error templates).
```

### 4.2 `/kimi:resume` (commands/resume.md)

Same shape as `continue.md`, except:
- `argument-hint: '<sessionId> <prompt>'`
- `$ARGUMENTS` first token parsed as sessionId (UUID); rest is prompt.
- Usage errors: empty `$ARGUMENTS` → "usage: /kimi:resume <sessionId> <prompt>"; single-token `$ARGUMENTS` → "missing prompt".

### 4.3 `/kimi:ask` modifications (commands/ask.md)

- `argument-hint` frontmatter: change `[--model <model>] [--resume <sessionId>] <prompt>` → `[--model <model>] <prompt>`.
- `### Options` section: delete the `--resume <sessionId>` bullet.
- No other content changes.

### 4.4 `lib/sessions.mjs` public API

```js
// Sync + fs-reads only. No network, no shell, no persistent state.

import { createHash } from 'node:crypto';
import { readFileSync, existsSync, statSync } from 'node:fs';
import { basename, join } from 'node:path';
import { homedir } from 'node:os';
import { errorResult } from './errors.mjs';

// Function-local homedir() (not a module-level constant) — see §8.1 S3:
// this makes HOME env overrides in tests work regardless of import order.
function kimiJsonPath() { return join(homedir(), '.kimi', 'kimi.json'); }
function sessionsDir()  { return join(homedir(), '.kimi', 'sessions'); }

/**
 * UUID regex — semi-strict 8-4-4-4-12 hex layout, no version-digit constraint.
 * Rejects degenerate inputs like "------------------------------------" (36 dashes)
 * or "00000000-0000-0000-0000-00000000000g" (non-hex at position 35) because
 * the regex enforces exact dash placement and hex-only character classes.
 * Probe v4 Q4.5 confirmed kimi-cli emits strict RFC 4122 v4; this regex accepts
 * strict v4 plus other valid UUID variants (v1, v3, v5, nil), so it is safe
 * against future kimi-cli UUID scheme changes while rejecting garbage.
 *
 * NOTE: this differs from lib/kimi.mjs:63's extraction regex `([0-9a-f-]{36})`
 * which is used in a contextual stderr parser (find-substring use case) where
 * false positives are tolerable. For our input-validation use case we want
 * tighter rejection of non-UUID input.
 */
const UUID_RE = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;

/**
 * Sanitize a cwd-basename value for safe inclusion in user-facing stderr strings.
 * Strips ANSI escape codes + ASCII control characters + newlines.
 * Prevents terminal-control-sequence injection via hostile cwd names (e.g.,
 * `mkdir $'cleared\e[2J\e[H'` — MiniMax finding, 6-way round 1).
 */
function sanitizeForStderr(s) {
  if (s == null) return '?';
  // Strip CSI, OSC, other ANSI escapes + ASCII control (incl. \n\r\t) + raw ESC.
  return String(s).replace(/\x1b\[[0-9;?]*[a-zA-Z]/g, '').replace(/[\x00-\x1f\x7f]/g, '');
}

/**
 * Compute the md5-hashed cwd path that kimi-cli uses for session directories.
 * Consumer passes a realpath-normalized cwd; see §7.7 for why we normalize
 * both sides (wrapper-side consistency workaround for kimi-cli's verbatim-
 * path storage per probe v3 `probe-results.json:47`).
 * Exposed for testability.
 */
export function md5CwdPath(normalizedCwd) {
  return createHash('md5').update(normalizedCwd).digest('hex');
}

**Design properties:**
- **Documented outcomes** (`kimi-json-missing` / `kimi-json-malformed` / `no-work-dir` / `no-last-session` / `invalid-uuid` / `session-not-found` / `session-empty` plus `ok:true`) are explicit `{ok, reason}` objects. Implementation uses internal try/catch so these never throw.
- **Unexpected FS errors** (`EACCES`, `EPERM`, `ENOTDIR`, `ELOOP`, dangling symlink dereference, etc.) are caught at the outer boundary and returned as `{ok:false, reason:'fs-error', detail}` — **not** thrown up the call stack. This preserves the "consumer always gets a structured outcome" contract without silently swallowing unexpected state.
- No I/O beyond fs reads. No directory writes, no process invocations.
- No `homedir()` captured into module-level constants — `kimiJsonPath()` and `sessionsDir()` re-read `$HOME` on each call, so tests that swap `process.env.HOME` after import still work.
- Consumer (`kimi-companion.mjs`) calls `mapSessionReason(r.reason, ctx)` on non-ok outcomes, writes the returned `errorResult.error` string to stderr, and calls `process.exit(errorResult.status)`.

## 5. Data flow (end-to-end)

### 5.1 `/kimi:continue <prompt>` happy path

```
User types /kimi:continue <prompt>
  ↓
Claude Code invokes commands/continue.md's Bash line:
  KIMI_COMPANION_CALLER=claude node kimi-companion.mjs continue "$ARGUMENTS"
  ↓
companion main() dispatches → case 'continue':
  ↓
(a) prompt = args.join(' ').trim()     — empty → stderr.write + process.exit(2)
(b) realCwd = resolveRealCwd()         — P3 lib/paths.mjs; realpath-normalized
(c) r1 = resolveContinueTarget(realCwd)
    reads ~/.kimi/kimi.json; parses JSON; iterates data.work_dirs (array) to find
    entry matching .path === realCwd; returns that entry's last_session_id or
    { ok:false, reason } if no match / no last_session_id / file missing / malformed
    ok=false → mapSessionReason → stderr.write(er.error) + process.exit(er.status)
(d) r2 = validateResumeTarget(realCwd, r1.sessionId)
    — double-checks disk consistency in case kimi.json is stale (e.g., user manually
      rm'd the session dir after kimi-cli recorded it as last)
    — checks: dir ~/.kimi/sessions/<md5(realCwd)>/<sessionId>/ exists; its
      context.jsonl exists AND size > 0 (probe v4 Q4.2 predicate)
    ok=false → mapSessionReason → stderr.write(er.error) + process.exit(er.status)
(e) result = await callKimi({ prompt, resumeSessionId: r1.sessionId })
    — existing callKimi path from lib/kimi.mjs:505; adds '-r <sessionId>' to kimi-cli args
    — prompt transport: callKimi internally chooses '-p <str>' for small prompts
      and '--input-format text + stdin pipe' for large prompts (threshold governed
      by lib/kimi.mjs#useStdinForPrompt at line 399). P2 does not alter this.
(f) hookTimingForResult(result)
    — P1 helper; flushes { timings, exitCode, signal } into timing store
(g) render stdout: response verbatim + footer '(session: <uuid> · model: <name> ...)'
(h) exit 0
```

### 5.2 `/kimi:resume <sessionId> <prompt>` happy path

Same as §5.1 but skips step (c); step (a) parses `<sessionId> <prompt>` as two tokens. `validateResumeTarget` then runs directly against the user-supplied `sessionId`. Step (e) still passes `resumeSessionId: <validated>` to `callKimi`.

### 5.3 Error path

Any failure in (a)/(c)/(d) → companion exits non-zero with stderr-line `Error: ...` matching §6 templates. No `kimi -r` invocation ever happens on an unvalidated target, which is the primary safety property.

### 5.4 Timing hook integration

Both branches call `hookTimingForResult(result)` at step (f). `result` from `callKimi` includes `{ timings, exitCode, signal }` per P1's P1 T9 instrumentation. Timing records are persisted before the companion returns, so `/kimi:timing --last` immediately reflects the resume call.

Hooking for error paths (where `callKimi` never ran): **no hook** — there is no LLM call to time. errorResult flows out directly.

## 6. Error handling

### 6.1 Literal error templates

**Path-exposure policy (§S1)**: user-facing strings use sanitized `<cwd-basename>` (last path component, stripped of ANSI escape codes and control characters per the `sanitizeForStderr` helper in §4.4) instead of the absolute path, to reduce accidental leakage of private filesystem layout and block terminal-control-sequence injection via hostile cwd names (MiniMax round-1 finding). The full `<realCwd>` still lives in the `errorResult.detail` field for debugging.

**v2 terminology fix**: the `session-empty` reason in v1/v1.1/v1.2 used the phrase "refusing to resume ghost session". Probe v4 Q4.4 proved that after a ghost has been materialized, its on-disk shape is identical to a real session — so calling out "ghost" in a user-facing message aimed at the *narrow* "dir exists but context.jsonl is absent or zero-byte" state is misleading (that state is NOT the post-probe ghost state; see §7.5 for the refined scope). v2 reword: "session has no stored messages; cannot resume".

**Exit code convention:**
- `2` = usage/syntax error (user needs to fix invocation — `/kimi:resume` with a malformed sessionId is the only v2 case)
- `1` = session-state or environment error (pre-validation refuses to invoke kimi-cli; user needs to fix state)
- **Note on `invalid-uuid` exit code** (Gemini round-1): for `/kimi:continue`, the UUID comes from kimi.json (not user input) so `invalid-uuid` there would mean kimi.json is corrupted — we redirect to exit 1 (treated as environment error, reported as if `kimi-json-malformed`). For `/kimi:resume` the UUID is user input so exit 2 is correct usage-error signalling. Implementation uses the exit-column-by-command form shown above.

This matches the existing convention in `/kimi:rescue` for `2` = usage error (rescue.md line 61).

### 6.3 Error-shape integration with P3 `errorResult()`

**Factual `errorResult()` contract** (from `plugins/kimi/scripts/lib/errors.mjs:16`):

```js

`errorResult()` is a **plain object builder**. It does NOT:
- print to stderr or stdout on its own
- call `process.exit()`
- apply an exit-code convention

Routing the returned object (stdout JSON vs. stderr plain-text) and the actual `process.exit(<status>)` call are the calling branch's responsibility. Existing usage pattern examples live at `kimi-companion.mjs:427,481,536,607,653,664`.

P2 adds per-reason templates + a builder, both co-located in `lib/sessions.mjs` (§N1) so that `errors.mjs` stays provider-agnostic and does not learn session-domain semantics:

```js
// lib/sessions.mjs additions — templates verbatim from §6.1 table
const SESSION_ERROR_REASONS = {
  'no-work-dir':         { status: 1, template: 'Error: no prior kimi session for this directory ({cwdBase}). Use /kimi:ask to start one.' },
  'kimi-json-missing':   { status: 1, template: 'Error: no prior kimi session for this directory ({cwdBase}). Use /kimi:ask to start one.' },
  'no-last-session':     { status: 1, template: 'Error: no prior kimi session for this directory ({cwdBase}). Use /kimi:ask to start one.' },
  'kimi-json-malformed': { status: 1, template: 'Error: ~/.kimi/kimi.json is malformed; cannot resolve last session.' },
  'session-not-found':   { status: 1, template: 'Error: session {sessionId} not found for this directory ({cwdBase}).' },
  'session-empty':       { status: 1, template: 'Error: session {sessionId} has no stored messages; cannot resume.' },
  'invalid-uuid':        { status: 2, template: 'Error: invalid sessionId format; expected UUID.' },
  'fs-error':            { status: 1, template: 'Error: filesystem access failed — {errCode}. Check permissions on ~/.kimi/.' },
};

export function mapSessionReason(reason, ctx = {}, options = {}) {
  // options.commandOrigin: 'continue' | 'resume' — affects exit status for
  // invalid-uuid (see §6.1 note). Default: 'resume' (exit 2 for invalid-uuid).
  const spec = SESSION_ERROR_REASONS[reason];
  if (!spec) {
    return errorResult({
      kind: 'sessions',
      error: `Error: internal — unknown session reason '${sanitizeForStderr(reason)}'.`,
      status: 1,
      detail: { reason, ctx }
    });
  }
  // For /kimi:continue, invalid-uuid means kimi.json has a malformed UUID —
  // this is an environment error (exit 1), not a user usage error (exit 2).
  let status = spec.status;
  if (reason === 'invalid-uuid' && options.commandOrigin === 'continue') {
    status = 1;
  }
  // Sanitize context values against ANSI/control-char injection (MiniMax round-1).
  const cwdBase = ctx.realCwd != null ? sanitizeForStderr(basename(String(ctx.realCwd))) : '?';
  const sessionId = ctx.sessionId != null ? sanitizeForStderr(String(ctx.sessionId)) : '?';
  const errCode = ctx.errCode != null ? sanitizeForStderr(String(ctx.errCode)) : '?';
  const filled = spec.template
    .replace('{cwdBase}', cwdBase)
    .replace('{sessionId}', sessionId)
    .replace('{errCode}', errCode);
  return errorResult({ kind: 'sessions', error: filled, status, detail: { reason, ...ctx } });
}
```

**Null-safety rationale (v2)**: `ctx.realCwd` could theoretically be null/undefined if a caller bug passes bad context. Previous v1.2 spec relied on `basename(null)` which Node returns as literal `'null'` string. v2 uses explicit `!= null` check producing `'?'` fallback. Same for `sessionId` and `errCode` templating.

**Origin-aware exit status (v2)**: `commandOrigin` lets mapSessionReason distinguish "UUID came from user input" (exit 2 usage error) vs "UUID came from kimi.json" (exit 1 environment error) — per §6.1 convention.

**Consumer (kimi-companion.mjs)** pattern — same as existing branches:

```js
const er = mapSessionReason(r.reason, { realCwd, sessionId });
process.stderr.write(er.error + '\n');
process.exit(er.status);
```

All P2 stderr output goes through this mapping for consistency with kimi-companion.mjs existing error-handling convention. No changes to `lib/errors.mjs`.

## 7. Edge cases decision matrix

### 7.1 `/kimi:continue` — cwd never used with kimi

**State**: No entry in `work_dirs` (array) matches both `.path === realCwd` AND `.kaos === 'local'`, or `kimi.json` itself does not exist.
**Behavior**: Hard fail. Reason `no-work-dir` or `kimi-json-missing`. Exit 1. Same user-facing message either way.

### 7.2 `/kimi:continue` — stale kimi.json (session rm'd)

**State**: matching `work_dirs[*]` entry has `last_session_id = U`, but `~/.kimi/sessions/<md5(realCwd)>/<U>/` no longer exists (user manually removed).
**Behavior**: Hard fail at step (d) of §5.1. Reason `session-not-found`. Exit 1. The stale kimi.json is **not** auto-healed by P2 (would require a write); users can run any `/kimi:ask <prompt>` to create a fresh session which kimi-cli will record as new `last_session_id`.

### 7.3 `/kimi:resume <id>` — id exists under a different cwd

**State**: `~/.kimi/sessions/<md5(realCwd)>/<id>/` does not exist, but `~/.kimi/sessions/<md5(otherCwd)>/<id>/` does.
**Behavior**: Hard fail. Reason `session-not-found`. Cross-cwd resume is **explicitly disallowed** (see §1.2). The error message intentionally says "not found for `<realCwd>`" rather than leaking the fact that the session exists elsewhere — the information is not useful at the CLI level and leaking it invites "why doesn't this just work?" support questions.

### 7.4 `/kimi:resume <id>` — malformed UUID (v2 regex refinement)

**State**: First token does not match the semi-strict regex `/^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i` (§4.4).
**Behavior**: Hard fail. Reason `invalid-uuid`. Exit 2 for `/kimi:resume` (user supplied the bad UUID → usage error); exit 1 for `/kimi:continue` if ever triggered (UUID came from corrupted kimi.json → environment error, §6.1). Declines to perform any disk I/O for invalid input. Regex rejects degenerate inputs like 36 dashes or 36 letters-with-g; accepts any UUID variant (v1/v3/v4/v5/nil) because probe v4 Q4.5 confirmed kimi 1.37 emits strict v4 but we don't lock to it for future-proofing.

### 7.5 `/kimi:resume <id>` — session-empty (v2 narrowed scope)

**State**: Directory exists at `~/.kimi/sessions/<md5(normalizedCwd)>/<id>/` but `context.jsonl` is: (a) absent (ENOENT), (b) a directory not a file, or (c) a regular file with size 0. v1.x described this as "ghost session refusal"; v2 narrows per probe v4 Q4.4 + Kimi source-read.
**Rationale for the narrow framing**: probe v4 Q4.4 demonstrated that once kimi-cli has run `-r <any-uuid>`, the session dir is structurally indistinguishable from a real session — so `session-empty` is NOT the post-hoc ghost-detection reason. Ghost is prevented at the `validateResumeTarget` existence check BEFORE invoking `kimi -r`. `session-empty` is a safety net for rare transient/corrupted states; probe did not observe kimi-cli producing such states in normal operation. MiniMax round-1 flagged a deliberately-rigged case (FIFO at context.jsonl path — see §7.8); the `isFile()` check in the implementation sketch covers it.
**Kimi source-read divergence note**: kimi-cli's own `Session.is_empty()` checks `wire_file.is_empty() AND context scan has no non-system roles`. Our `size > 0` predicate is stricter in a benign direction — any bytes count as populated. Benign divergence; documented here and in §4.4.
**Behavior**: Hard fail. Reason `session-empty`. Declines to invoke `kimi -r`.

### 7.6 Concurrent modification of kimi.json (v2 framing upgrade)

Two distinct race scenarios are acknowledged:

**6.1 Malformed-read race** (v1 original): kimi-cli writes to kimi.json mid-read from wrapper. Strict `JSON.parse` fails → `kimi-json-malformed`. User can retry. No wrapper-side lock (kimi-cli owns writes). Accept-the-race; worst case is one spurious error. Do NOT fall back to permissive parsing.

**6.2 Wrong-session-resume race** (v2 upgrade, MiniMax round-1): wrapper reads `last_session_id = U1`, kimi-cli atomically rewrites to `U2`, wrapper validates U1 (still exists on disk — kimi-cli never deletes old sessions), wrapper invokes `kimi -r U1`. User's newer context (U2) is NOT resumed; U1 is. No wrapper or kimi-cli error. **User sees silent wrong-session resume.** This is a known limitation. Mitigations considered and rejected in v2:
- File-lock kimi.json: conflicts with kimi-cli's write-side ownership; kimi-cli likely uses atomic-rename rather than advisory lock, so cross-side locking is not coherent. High complexity.
- Double-read after validate: narrows the window but does not close it (the `kimi -r` invocation is still unlocked). Partial mitigation.
- Refuse if another kimi process detected: too aggressive; breaks legitimate multi-terminal usage.
**Test posture** (Qwen round-1): this race is **observational only; no deterministic test is attempted**. Reproducing it would require precise timing control over the concurrent kimi-cli write and is not worth the fixture complexity. Known limitation; documented so an incident investigator can recognize the signature.

### 7.7 cwd path normalization — realpath-vs-verbatim (v2 corrected narrative)

**Observed state (probe v4 Q4.1, kimi 1.37)**: `work_dirs[].path` stores realpath-resolved paths (e.g., `/private/tmp/foo` on macOS when input was `/tmp/foo`).

**Observed state (probe v3, kimi 1.36)**: `work_dirs[].path` was reported as storing verbatim paths (symlinks NOT resolved) per probe 06 `path_storage_format: "verbatim-absolute-normalized"`.

**Source-read reconciliation** (Kimi round-1): there is NO deliberate kimi-cli code change between 1.36 and 1.37 that produces this difference. `kimi_cli/kaos/path.py::canonical()` still only does dot-segment removal (no symlink resolution) in 1.37. The observed "realpath delta" is actually **platform `Path.cwd()` behavior** — on macOS, `Path.cwd()` returns the symlink-resolved current directory because the kernel tracks real cwd. So when probe v4 ran kimi from `/tmp/kimi-probe-v4-*`, kimi's internal `KaosPath.getcwd()` saw `/private/tmp/kimi-probe-v4-*` because the kernel had already resolved the symlink. Probe v3's 2026-04-20 test may have used a cwd that wasn't under a symlink path, so `canonical()` passed through unchanged — hence the "verbatim" observation. Both observations are consistent with the source code; the platform is the hidden variable.

**Decision**: P2 wrapper calls `resolveRealCwd()` (P3 `lib/paths.mjs`) on **both** sides — the cwd passed to `resolveContinueTarget` + `validateResumeTarget` AND the `cwd` argument passed to `callKimi` (§4.5). This establishes symmetry regardless of how kimi-cli internally canonicalizes.

**Why this works**: if our md5 is computed on `resolveRealCwd(cwd)`, and kimi-cli computes its md5 on `canonical(Path.cwd())` where `Path.cwd()` returns the realpath-resolved cwd (macOS default), both md5s match. For shells that don't resolve cwd (Linux with non-symlinked cwd), `resolveRealCwd()` is a no-op, and kimi-cli's `canonical()` is also a no-op on already-absolute paths — still symmetric.

**Behavior**: Always call `resolveRealCwd()` before touching sessions AND before passing cwd to `callKimi`. §4.5's `cwd: realCwd` parameter to callKimi is the enforcement point.

### 7.8 Filesystem anomalies (v2 corrected + expanded)

Unusual filesystem state that should not crash the wrapper:

| State | Behavior |
|---|---|
| `process.env.HOME` unset | `os.homedir()` falls back to the `passwd` struct's home for the current uid (e.g., `/Users/<login>` on macOS for regular user, `/var/root` only when effective uid is root). kimi-json-missing or no-work-dir then fires normally. **Not** a `/var/root` fallback for regular users — v1.2 spec said otherwise but was incorrect (Gemini round-1 nitpick upgraded). |
| `~/.kimi/kimi.json` is a symlink (resolvable) | `readFileSync` follows the symlink; normal path. |
| `~/.kimi/kimi.json` is a dangling symlink | `readFileSync` throws `ENOENT` → `kimi-json-missing`. |
| `~/.kimi/sessions/<md5>/<uuid>/` is a dangling symlink | `statSync` throws `ENOENT` → classified as `session-not-found` (NOT fs-error — MiniMax round-1). From the user's perspective, a dangling symlink at session-dir level is structurally equivalent to "no session dir there". |
| `~/.kimi/sessions/<md5>/<uuid>/` is a symlink loop | `statSync` throws `ELOOP` → wrapped as `fs-error` with detail `errCode: 'ELOOP'`. |
| `~/.kimi/sessions/<md5>/<uuid>/` owned by different uid, not readable by current user | `EACCES` on `statSync` or `readdirSync` → returned as `fs-error` with detail. |
| `~/.kimi/sessions/<md5>/<uuid>/context.jsonl` is a FIFO/pipe | `statSync` reports `.isFile() === false` for FIFOs, so the `!ctxStat.isFile()` branch in validateResumeTarget returns `session-empty`. Kimi is not invoked; the hostile pipe is not opened by us. (MiniMax round-1.) |
| `~/.kimi/sessions/<md5>/<uuid>/` is a regular file instead of a directory | `statSync(dir).isDirectory() === false` → returns `session-not-found` rather than attempting `statSync(dir/context.jsonl)` which would throw `ENOTDIR`. (Qwen round-1.) |
| `~/.kimi/sessions/<md5>/<uuid>/context.jsonl` is a directory | `statSync(ctx).isFile() === false` → returns `session-empty`. |
| `~/.kimi/kimi.json` contains trailing garbage after valid JSON | Strict `JSON.parse` rejects → `kimi-json-malformed` (§7.6). |
| `~/.kimi/sessions/<md5>/` does not exist at all (no prior kimi-cli use in this cwd) | For `/kimi:resume`: `statSync` on `<md5>/<uuid>` throws ENOENT → session-not-found. For `/kimi:continue`: resolveContinueTarget would have already failed at `no-work-dir`; this validate call never runs. |
| `~/.kimi` itself does not exist | `readFileSync(kimi.json)` throws ENOENT → `kimi-json-missing`. |
| Hostile ANSI escape codes or control chars in `realCwd` basename | `mapSessionReason` sanitizes via `sanitizeForStderr` (§4.4) before substituting into user-facing templates. Raw escape codes are stripped. (MiniMax round-1.) |
| `~/.kimi/kimi.json` parses to `null` or a non-object value | Returns `kimi-json-malformed` with detail `'top-level not an object'` (§4.4 implementation sketch has the explicit check). (Gemini round-1.) |

## 9. Breaking changes + migration

### 9.1 BREAKING: `/kimi:ask --resume <id>` removed

**What changes**: The `--resume <sessionId>` flag is deleted from `/kimi:ask` (both `argument-hint` frontmatter and the companion's option parser). Any existing invocation fails with "unknown flag" exit 2.

**Migration path**: `/kimi:ask --resume <uuid> <prompt>` → `/kimi:resume <uuid> <prompt>`. Exact same underlying behavior — `callKimi` is still invoked with `resumeSessionId: <uuid>` — plus the new validation guarantee.

**Affected users**: anyone who memorized the `/kimi:ask --resume` shape from v0.1. Spec-level expectation: low impact (sibling plugins never had this flag — see §0.3 scan paths; kimi docs have not heavily promoted it).

### 9.2 BREAKING: ghost-session path hardened

**What changes**: invocations that previously appeared to succeed with bogus UUIDs (probe v4 Q4.0 confirmed: kimi `--print -r <fab-uuid>` silently creates a fresh session at the fabricated UUID with exit 0 and full 3-file session dir) now hard-fail with `session-not-found` or `session-empty` before reaching kimi-cli.

**Migration path**: verify UUIDs; use `/kimi:continue` for "keep going" intent, `/kimi:resume` only when you have the right id.

**Affected users**: anyone whose existing scripts pass UUIDs that silently drift to ghost sessions. Failure mode is loud and safe (no silent data loss; user sees clear error).

### 9.3 PR-split trade-off (S4)

**Decision**: new commands + `/kimi:ask --resume` removal ship in the **same PR** (user-selected A1 during brainstorm).

**Arguments for the same-PR choice (chosen)**:
- Sibling-backport clarity: the entire "resume-UX overhaul" lands as one unit; future gemini/minimax/qwen backports follow a single diff.
- Single CHANGELOG entry telling the full story instead of two partial stories.
- Test coverage for the removal lives in the same `tests/commands-p2.test.mjs` as the new commands, avoiding a standalone test file for a one-line parser removal.

**Arguments against (considered and rejected)**:
- **Attribution risk on regression**: if a bug surfaces post-merge, bisect can attribute it to "the P2 commit" but isolating cause between "new command path" vs. "ask.md option-parser change" takes an extra read. Counter: git-blame on the specific line is sufficient; this is a minor forensic inconvenience, not a blocker.
- **Review surface**: a same-PR change is larger. Counter: the ask.md delta is ~5 lines total (frontmatter hint + option parser arm + pass-through), dwarfed by the new-command scaffolding.

**Conclusion**: A1 (same PR) remains the right call. Documented here so future spec reviewers or a post-merge incident investigator can see the trade-off was considered, not overlooked.

Reviewers: is there value in tightening to v4 pattern for a slightly better error message, or stay permissive for future-proofing? Low-stakes — spec stays permissive unless strongly argued.
2. **kimi.json concurrent-write window**: §7.6 accepts a race where a brief read during kimi-cli write may return `kimi-json-malformed`. Should we retry-once with a 50ms delay before surfacing the error? Tradeoff: extra latency on happy path (milliseconds) vs. rare spurious error (seconds of user confusion). Weak preference: no retry — a retry masks the rare real case where kimi.json is persistently malformed, and users can manually retry.
3. **Ghost-session counter-proposal**: should P2 additionally *delete* detected ghost sessions (`rm -rf <ghost-dir>`) as a courtesy cleanup, or strictly stay read-only? Current spec: strictly read-only (refuse + do not delete). Probe v4 Q4.4 showed ghost and real are post-hoc identical, so there's no reliable way to detect "past ghost" we could safely clean up — deletion would be risky. Sticking with read-only.
4. **`--model` flag parity**: if a user wants to resume session X but try a different model, current spec says defer. Is deferring OK, or do users request this parity with `/kimi:ask --model`? v0.2 evidence: no one asked yet. YAGNI stays.
5. **fs-error granularity**: current spec lumps EACCES / EPERM / ENOTDIR / ELOOP / dangling symlink all under single `fs-error` reason. Should any of these split out (e.g., a dedicated `permission-denied` reason with distinct user message)? Low-value — permissions on `~/.kimi/` are uniformly rare.
6. **[v2 UPDATE — KIMI SOURCE-READ RESOLVED] `/kimi:continue` design — explicit vs native `kimi -C`**: Kimi 6-way round-1 reviewer read `kimi_cli/cli/__init__.py:560-568` and confirmed that `kimi -C` in a cwd with no prior session calls `Session.continue_(work_dir)` which returns `None`, then raises `typer.BadParameter("No previous session found...")` — **exit 2, no session created, no ghost**. This is SAFE native behavior. The question is no longer "is -C risky?" but "do we prefer wrapper-controlled explicit path or native delegation?"
   - **Option A (current spec)**: `/kimi:continue` does explicit `resolveContinueTarget(cwd)` (read kimi.json, iterate work_dirs) → validate target → call `kimi -r <resolved-uuid>`. Fully wrapper-controlled error messages (our §6.1 templates, not kimi's), deterministic error paths, and same code path as `/kimi:resume` (one implementation, two entry points).
   - **Option B (alternative, safe-per-source-read)**: `/kimi:continue` does NO kimi.json parsing; just invokes `kimi -C -p <prompt>`. If no prior session, kimi emits its own BadParameter message; we pass it through. Simpler, relies on kimi-cli's own logic. Downside: error messages are kimi's (not our §6.1 templates), which diverges from `/kimi:resume`'s wrapper-side error vocabulary. Also: a future kimi-cli version could change `-C` behavior; spec would have an uncovered path.
   - **Decision for v2**: keep Option A. Rationale: (1) uniform error vocabulary across continue/resume; (2) `/kimi:continue` still needs wrapper-side `hookTimingForResult` — we're running through `callKimi` either way; (3) the duplicated work (parsing kimi.json when kimi could do it) is trivial compared to the LLM call. Option B would save ~5 lines of lib code but cost error-vocabulary divergence and a kimi-version-dependence we'd have to probe again on every upgrade. 6-way reviewers can still override this if they disagree.
7. **Tests for context.jsonl size > 0 predicate**: spec §4.4 `validateResumeTarget` uses `size > 0` — should we tighten to "valid JSONL (≥1 parseable line)" to catch whitespace-only files (MiniMax round-1 edge)? Probably overkill (tests cover zero-byte; whitespace-only file would be flagged by kimi-cli itself on resume). Reviewers: flag if you disagree.
8. **Wrong-session-resume race** (§7.6 v2 scenario 6.2, MiniMax round-1): the documented mitigation is "accept". Reviewers: is there a lightweight wrapper-side sanity check worth adding, like "after callKimi returns, compare the session id from kimi stderr's resume hint to the one we told it to resume — warn on mismatch"? Trade-off: extra post-validation step + mismatch-handling logic vs. silent-wrong-session-resume staying possible. v2 does NOT add this check (YAGNI until a real user hits it); surfaced for review feedback.
