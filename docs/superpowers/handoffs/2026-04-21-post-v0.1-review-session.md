## Methodology observations (for lessons.md §F or §H)

1. **Triple-source version numbers.** First time around I got `1.37.0` from the Moonshot docs-site changelog via WebFetch — but the fetcher uses a small/fast model for summarization, so the "1.37.0" content I surfaced was paraphrased, not verbatim. A parallel WebSearch also returned "PyPI latest is 1.30.0" (stale index). User caught both; cross-checking GitHub Releases + PyPI + Homebrew resolved: 1.37.0 is real, my summarized changelog was not verbatim. **Rule**: when two independent sources disagree on a version number, hit a third before writing it down.

2. **Self-review catches what review misses.** My initial H3 fix covered the 4 kimi-spawn entry points and I reported it as complete. When user said "先 review 再开 PR", a fresh re-read caught that `resolveWorkspaceRoot` was still called with raw cwd from 4 non-spawn sites — producing slug drift in non-git + symlinked tmpdirs. That was a real regression. **Rule**: for any multi-site change, the fresh-read pass must explicitly enumerate all call sites even when the initial fix "feels complete".

3. **Subagent summaries drift from source.** The two parallel Explore agents during the review stage produced high-quality findings — but several of their claims were false positives (e.g. "exit 124 missing from kimi-cli-runtime" was wrong; it's there at line 58). Always verify agent claims against raw source before reporting. The H1/H2/H3 findings survived verification; some Medium-level claims didn't.
